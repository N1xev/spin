<objective>
Wire the runner end-to-end: confirm `--list` and `--explain` show source labels, the source-precedence chain is stable, the cargo fallbacks from the rust ecosystem's `Tasks()` flow into the fallback source, and add unit-test coverage for the runner, params parser, and template loader. This delivers Phase 5 success criterion 4 and consolidates the cross-cutting test surface for the v2.0 skeleton.

Purpose: The runner's source chain is wired in `cmd/run.go` already (per STATE.md) but lacks ecosystem integration (the rust cargo fallbacks come from `rust.Tasks()`, not from hardcoded strings). The source chain also lacks unit-test coverage, and the params parser / template loader have no tests. Without this plan, the runner is a "looks-fine" layer that breaks silently when the ecosystem list grows.

Output: An upgraded `cmd/run.go` `defaultSourceChain` that injects ecosystem task maps; upgraded `internal/runner/{source,list,execute,explain}.go` for stable, well-tested behaviour; new test files for runner, params, and template loader. Build must remain green and all v1.0 commands must still work.

**Note on file naming:** the runner package's main source file is `internal/runner/source.go` (not `runner.go`). It contains the `Runner`, `Task`, `merge`, `Resolve`, and `All` symbols. All references in this plan use `source.go`.
</objective>

1. Create a new file `internal/runner/sources/ecosystem.go`:
   - Package `sources`. Defines a new source type `ecosystemTasks` that wraps the registered ecosystems' `Tasks()` method.
   - `NewEcosystemTasks(ecos []ecosystem.Ecosystem) runner.TaskSource` -- constructor.
   - `Name() string` returns `"ecosystem"`. `Order() int` returns 5 (just above the fallback's 0, but below scripts/).
   - `Detect(dir string) bool` returns `true` whenever any of the wrapped ecosystems' `Detector.Matches(dir)` returns true. Use the existing detector interface from `internal/ecosystem/ecosystem.go`.
   - `Tasks(dir string) ([]runner.Task, error)` returns the union of `eco.Tasks()` for every ecosystem that matches. Mark each task's `Source` field with `ecosystem:<name>` so `--list` output is unambiguous.
   - This is the AUTHORITATIVE source for language fallbacks. The hardcoded cargo fallbacks in `fallback.go` are now redundant for cargo -- but the v2.0 fallback stays (in case no ecosystem is registered), so we keep both. The ecosystem tasks source wins on conflict (higher Order).

   - Add `sources.NewEcosystemTasks(defaultRegistry().All())` to the chain.
   - The full chain becomes (in Order ascending, last in slice = highest precedence): fallback(0), ecosystem(5), scripts(20), packagejson(30), makefile(40), taskfile(60), spinconfig(100). Order in the slice does not matter; only `Order()` does. The runner's `merge` function (in `internal/runner/source.go`) uses `t.Order > cur.Order` to choose the winner, so higher Order wins. Ecosystem at Order=5 wins over fallback at Order=0 -- correct.
   - Update the comment in `defaultSourceChain` to reflect the new chain.

3. Create a new test file `internal/runner/sources/ecosystem_test.go` with one end-to-end test that exercises the full chain (this is the "verify the chain works end-to-end" check that was previously inlined in the action and would not have been run by `go test`):

Call `r.All()`, find the `build` task, assert its `Source` field contains `ecosystem:rust` (NOT `fallback`).
   - Also assert that the `clippy` and `fmt` tasks exist (these come ONLY from the ecosystem's `Tasks()`, not from the hardcoded fallback -- proving the ecosystem source is the authority for language fallbacks).
   - The test file imports `internal/ecosystems/rust` directly. This is a fresh test file in the `sources` package, not in `internal/ecosystem`, so it is safe to import a concrete ecosystem package here (no import cycle).
   - Use `t.TempDir()` for isolation. No new dependencies.

4. Manual smoke check (run by the executor after the test passes, but NOT part of the automated `go test` run):
   - In a scratch tempdir, write a `Cargo.toml` and run `./spin run --list`. Confirm the `build` task is labelled `ecosystem:rust`. This is documentation-only -- the automated test in step 3 is the load-bearing verification.
