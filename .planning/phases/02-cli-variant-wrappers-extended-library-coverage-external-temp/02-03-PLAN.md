# Plan 02-03: CLI variant + --all combo + 7 lib overlays (incl. --viper) + base go.mod/README updates

## Phase Goal

**As a** Go developer, **I want to** generate a working CLI or TUI project scaffolded with any charmbracelet v2 library, **so that** I can start building immediately without manual setup of dependencies, tooling, or build configs.

This plan is the user-visible content of Phase 2. It fills in the 4 stub variant/overlay template files Phase 1 left as forward-compat placeholders, adds 7 new lib overlays (`huh`, `glamour`, `glow`, `wish`, `log`, `harmonica`, `viper`), and extends `_base/go.mod.tmpl` so the generated project references all 11 charmbracelet libs (gated by `{{if hasX .}}` blocks). The end result: `spin new foo --cli --cobra --fang` produces a runnable CLI; `spin new foo --cli --cobra --fang --viper` produces a CLI with viper-wired config; `spin new foo --all --bubbletea --cobra --fang` produces a single binary with both TUI and CLI subcommands; `spin new foo --tui --huh --glamour --log --harmonica` produces a TUI with each lib wired into a working example.

This is the vertical-slice "user can scaffold a fully-featured project in one command" deliverable for Phase 2's first three success criteria (CLI variant works, all 7 new libs work, go.mod is complete).

## Context

Phase 1 shipped 3 active lib overlays (`bubbletea`, `bubbles`, `lipgloss`) and 1 TUI variant. Phase 1 forward-compat-stamped 13 bool fields on the `Project` struct (`Cobra`, `Fang`, `Viper`, `Huh`, `Glamour`, `Glow`, `Wish`, `Log`, `Harmonica`, plus 4 `charmbracelet/x` flags) and 13 corresponding `--<flag>` boolean flags on `cmd/new.go`, but the templates for those flags do not exist yet — `variant_cli/main.go.tmpl` and `variant_all/main.go.tmpl` are 3-line `// TODO` stubs, and the 7 new lib directories have only `LIBS.md.tmpl` placeholders. The `funcMap` in `template.go` has `hasCobra`, `hasFang`, and `requiresImport` (which can return `true` for huh/log), but no top-level `hasHuh`, `hasGlamour`, etc. (Plan 02-02 added those.)

The Walking Skeleton regression test in `internal/scaffold/integration_test.go` (8 assertions including `go 1.25.0`, `tea.NewProgram`, `lipgloss.NewStyle`, `.air.toml` `build.entrypoint`, `Taskfile.yml` `setup:`) must still pass — this plan's template changes are strictly additive.

For `--viper` specifically (FLAG-15), research §3.3 calls for a real `internal/config/config.go.tmpl` overlay that exports `Bind()` and `LogLevel()` functions, plus a `{{if hasViper .}}` block in `variant_cli/main.go.tmpl` that calls `config.Bind(rootCmd)` in `init()` and adds a `--log-level` persistent flag. The viper overlay is the most non-trivial of the 7 because it has real wiring in main.go (not just a standalone example function).

### Task 2: Replace `variant_all/main.go.tmpl` TODO with TUI + CLI combo (+ viper wiring)

**Files:** `internal/scaffold/templates/variant_all/main.go.tmpl`

**Action:** Replace the 3-line `// TODO` stub with a real runnable combo per research §3.2: a single binary with cobra root, a `tui` subcommand (bubbletea v2), and a `hello` subcommand (CLI), plus the same viper wiring as Task 1. Bare `myapp` invocation shows cobra's help (strict-subcommand mode per research Q7).

The template structure:

- `package main` with `// Code generated by spin {{.SpinVer}}. DO NOT EDIT.`
- Imports: `context`, `fmt`, `os`, `charm.land/bubbletea/v2`, `charm.land/fang/v2`, `github.com/spf13/cobra`, `{{if hasViper .}}{{.Module}}/internal/config{{end}}`
- Persistent flag `--log-level` (default `"info"`) — gated by `{{if hasViper .}}`
- `var rootCmd = &cobra.Command{Use: "{{.Name}}", Short: ..., Long: ...}`
- `var tuiCmd = &cobra.Command{Use: "tui", Short: "Launch the TUI", RunE: ...}` that calls `tea.NewProgram(initialModel()).Run()` and returns the error
- `var helloCmd = &cobra.Command{Use: "hello [name]", Args: cobra.MinimumNArgs(1), RunE: ...}` that prints `Hello, %s!\n`
- `type model struct { count int }` with `Init() tea.Cmd { return nil }`, `Update(msg tea.Msg) (tea.Model, tea.Cmd)` (handles `tea.KeyPressMsg` for q/esc/ctrl+c to quit, up/down to increment/decrement count), and `View() tea.View` (returns `tea.NewView(...)` with a counter display)
- `func initialModel() model { return model{} }`
- `func init() { rootCmd.AddCommand(tuiCmd, helloCmd); {{if hasViper .}}config.Bind(rootCmd){{end}} }`
- `func main() { fang.Execute(context.Background(), rootCmd, fang.WithVersion("{{.SpinVer}}")) }`

The combo intentionally keeps both the TUI runtime and the cobra root in a single file (no `internal/tui/` split) for the v1 simplicity. Phase 4's `spin doctor` may check for `cmd/<name>/main.go` restructure as a follow-on.

### Task 3: Create the 7 new lib overlay template files (incl. viper)

**Files:** `internal/scaffold/templates/lib/huh/huh.go.tmpl` (new, replaces LIBS.md.tmpl), same for glamour, wish, log, harmonica, glow, viper

**Action:** For each of the 7 libs, replace the `LIBS.md.tmpl` placeholder with a real `.go.tmpl` (or `.md.tmpl` for glow, which is a binary not a Go lib) overlay that provides a small, standalone, runnable example. The overlay file is `package main` (lives at the project root next to `main.go`); the example function is named after the lib for easy reference.

Per research §6, the example snippets are:

- **huh** (`lib/huh/huh.go.tmpl`): `func huhExample()` runs a 2-field `huh.NewForm` with an `huh.NewInput` and `huh.NewConfirm`. The function is invoked from the variant_tui/main.go.tmpl when the user presses `?` (Task 4).
- **glamour** (`lib/glamour/glamour.go.tmpl`): `func glamourExample() string` returns a rendered markdown snippet via `glamour.Render("## hello", "dark")`. Surfaces in main.go when the user presses `?` for help.
- **wish** (`lib/wish/wish.go.tmpl`): `func wishExample()` calls `wish.NewServer(...)` with `bubbletea.Middleware`, `activeterm.Middleware`, `logging.Middleware`. Surfaces as an `ssh` subcommand in variant_tui/main.go.tmpl when `--wish` is set. (For v1, the variant_tui wiring is just a stub; the user extends it.)
- **log** (`lib/log/log.go.tmpl`): `func logSetup()` calls `log.SetDefault(log.New(os.Stderr))` and `log.SetLevel(log.InfoLevel)`. Surfaces as a `setupLogger()` call in main().
- **harmonica** (`lib/harmonica/harmonica.go.tmpl`): `func harmonicaExample()` runs a 60-frame spring animation from 0.0 to 100.0 with `harmonica.NewSpring(harmonica.FPS(60), 6.0, 0.5)`. Surfaces as a TUI keybinding on `s`.
- **glow** (`lib/glow/README.glow.md.tmpl`): a sample markdown file that glow renders well. The Taskfile.yml `setup` target already installs glow; the README in `_base/` mentions `glow README.md` (added in Task 5). Glow is a binary, no Go import.
- **viper** (`lib/viper/internal/config/config.go.tmpl`): a `package config` file (NOT `package main`) that exports `Bind(root *cobra.Command) error` and `LogLevel() string`. Per research §3.3, `Bind` does `viper.BindPFlag("log-level", root.PersistentFlags().Lookup("log-level"))` + `viper.SetConfigName(".myapprc")` + `viper.SetConfigType("yaml")` + `viper.AddConfigPath(".")` + `viper.ReadInConfig()` (ignoring `viper.ConfigFileNotFoundError`) + `viper.SetEnvPrefix("MYAPP")` + `viper.AutomaticEnv()`. `LogLevel()` returns `viper.GetString("log-level")` or `"info"` if not set. Because this is a separate package, the generated `main.go` adds an import `"{{.Module}}/internal/config"` and calls `config.Bind(rootCmd)` in `init()` (Task 1, Task 2).

Each lib overlay file must compile standalone (the project builds with CGO_ENABLED=0 even when the example function is unused; Go's "imported but not used" rule applies to imports, not unused functions, so a `func huhExample()` that no one calls is fine). The viper overlay is the exception: it must be imported (because `package config` has a non-`init` body that the main package references), and the main.go template must reference it when `hasViper` is true.

The overlays also have a `// generated by spin {{.SpinVer}}. DO NOT EDIT.` header for traceability (matches the existing variant_tui/main.go.tmpl pattern).

**Verify:** Integration tests `TestIntegrationScaffold_<Lib>` for each of the 7 libs (added in Task 8) run the end-to-end smoke and assert:
- The lib's `require` line is in `go.mod`
- The lib's overlay file exists at the project root (or `internal/config/config.go` for viper)
- `go build ./...` and `go test ./...` both exit 0
- The generated `main.go` references the lib (when wired in via variant_tui's `{{if hasX .}}` block; for viper, that `internal/config/config.go` is imported in main.go and `config.Bind(rootCmd)` appears in `init()`)

**Commit:** `feat(02): add 7 lib overlays (huh, glamour, wish, log, harmonica, glow, viper)`

## Anti-patterns

- Do NOT add the 7 new libs to `_base/go.mod.tmpl` without `{{if hasX .}}` gates — every scaffolded project should only require the libs it actually uses. A bare `spin new foo --tui --bubbletea` should NOT pull in wish (an SSH server framework with 1.25.9 floor).
- Do NOT remove the `{{if hasCobra .}}` or `{{if hasFang .}}` gates for cobra/fang — the CLI variant uses them but the TUI variant should not pull in cobra (the TUI variant's main.go does not use cobra).
- Do NOT wire `wish` into the TUI variant as a keybinding — wish needs a host key path and an SSH server lifecycle; the right pattern is an `ssh` subcommand. Defer the full wiring to a future plan; v1 just has a comment in main.go.
- Do NOT inline the 6 new lib overlay content into `variant_tui/main.go.tmpl` — the overlay engine's job is to keep them in `lib/<name>/` so users can extend or replace them without forking main.go.
- Do NOT bump the spin binary's own go.mod (it stays at 1.25.8 from Phase 1). Only the generated `go.mod` changes.
- Do NOT add `--glow` as a `{{if hasGlow .}}` block in `_base/go.mod.tmpl` — glow is a binary, not a Go import. The README mention is enough for v1.
- Do NOT add a `{{if hasViper .}}` block in `variant_tui/main.go.tmpl` — viper is a config concern, not a TUI concern. The viper overlay at `lib/viper/internal/config/config.go.tmpl` is created regardless, but only the CLI variants reference it.
- Do NOT use `urfave/cli` or any other CLI framework in the viper overlay — the existing `cobra` + `viper.BindPFlag` pattern from research §3.3 is the canonical approach.
