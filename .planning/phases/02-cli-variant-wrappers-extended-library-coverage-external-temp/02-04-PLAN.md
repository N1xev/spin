# Plan 02-04: Five wrapper subcommands + extended CI grep suite + wrapper integration tests

## Phase Goal

**As a** Go developer, **I want to** generate a working CLI or TUI project scaffolded with any charmbracelet v2 library, **so that** I can start building immediately without manual setup of dependencies, tooling, or build configs.

This plan ships the second half of Phase 2's user-facing value: 5 wrapper subcommands (`spin run`, `spin build`, `spin test`, `spin vet`, `spin fmt`) that wrap the Go toolchain with one tool, plus 2 new CI grep scripts (`check-air-bin.sh`, `check-taskfile-setup.sh`) that gate regressions in the generated `.air.toml` and `Taskfile.yml`. The wrappers are pure scaffolder-side (they don't touch the generated project); the grep scripts gate the templates that 02-03 and prior plans produce.

Each wrapper follows the same `ToolSpec` + `RunWithFallback` pattern (research §4.2): the helper looks up the preferred tool on `$PATH`, runs it; if missing, prints a one-line install hint and falls back to stock Go. The only wrapper with unique logic is `spin test` (prism needs Go 1.24+; the detector checks both Go version and `$PATH` presence).

## Context

Phase 1 shipped `cmd/new.go` (the `spin new` subcommand) and `cmd/root.go` (the cobra root). Phase 2 adds 5 more subcommands attached to the same root via the `init()` pattern. The shared `ToolSpec` + `RunWithFallback` helper in `internal/wrap/detect.go` is the design center -- 5 callers, one helper.

The CI grep suite from Phase 1 has one script (`scripts/check-v1-leaks.sh`) that catches v1 charmbracelet API leaks. Phase 2 splits the air-config check into its own script and adds a new Taskfile-setup check. The three scripts run together via the `Taskfile.yml` `grep-v1-leaks` target.

The 41 Phase 1 tests + 9 Plan 02-03 tests must still pass -- this plan's wrapper subcommands don't touch the scaffolder's template engine; they only add new files in `cmd/` and `internal/wrap/`. The Walking Skeleton scaffold (`--tui --bubbletea`) is unchanged.

### Task 1: Create `internal/wrap/detect.go` with `ToolSpec` + `RunWithFallback`

**Files:** `internal/wrap/detect.go` (new)

**Action:** Create the new package with the shared helper per research §4.2:

```go
type ToolSpec struct {
    Name        string   // e.g. "prism"
    Args        []string // passed to exec.Command
    ExtraEnv    []string // e.g. CGO_ENABLED=0
    InstallHint string   // "go install go.dalton.dog/prism@latest"
}

func RunWithFallback(spec, fallback ToolSpec) error {
    if path, err := exec.LookPath(spec.Name); err == nil {
        return runTool(path, spec.Args, spec.ExtraEnv)
    }
    fmt.Fprintf(os.Stderr, "hint: %s not found on $PATH; %s\nfalling back to: %s\n",
        spec.Name, spec.InstallHint, fallback.Name)
    return runTool(fallback.Name, fallback.Args, fallback.ExtraEnv)
}

func runTool(name string, args, extraEnv []string) error {
    cmd := exec.Command(name, args...)
    cmd.Stdin = os.Stdin
    cmd.Stdout = os.Stdout
    cmd.Stderr = os.Stderr
    if len(extraEnv) > 0 {
        cmd.Env = append(os.Environ(), extraEnv...)
    }
    return cmd.Run()
}
```

The package's `doc.go` (or top-of-file comment) explains that the `ToolSpec` + `RunWithFallback` design is the single helper for all 5 wrappers, and that `runTool` is unexported because it's an implementation detail.

**Verify:** `go build ./...` exits 0. Unit test `TestRunWithFallback_PreferredExists` and `TestRunWithFallback_PreferredMissing` in Task 8.

**Commit:** `feat(02): add internal/wrap package with ToolSpec + RunWithFallback`

```go
func Build() error {
    cwd, err := os.Getwd()
    if err != nil { return err }
    name := filepath.Base(cwd)
    if err := os.MkdirAll("bin", 0o755); err != nil { return err }
    spec := ToolSpec{
        Name: "go",
        Args: []string{"build", "-o", filepath.Join("bin", name), "."},
        ExtraEnv: []string{"CGO_ENABLED=0"},
        InstallHint: "go is in your Go toolchain",
    }
    // No fallback (go build is the only path); use the spec directly.
    return runTool(spec.Name, spec.Args, spec.ExtraEnv)
}
```

`build` has no upgrade path beyond `go build` itself; the wrapper exists to give the user a uniform `spin build` command and to wire the `CGO_ENABLED=0` env automatically.

### Task 4: Create `internal/wrap/test.go` (prism with Go 1.24+ detector)

**Files:** `internal/wrap/test.go` (new)

**Action:** Implement `Test() error` per research §4.4 with the prism + Go version detector:

```go
func Test() error {
    goOld := goVersionLessThan("1.24")
    if !goOld {
        if path, err := exec.LookPath("prism"); err == nil {
            spec := ToolSpec{Name: path, Args: []string{"go", "test", "./..."}}
            return runTool(spec.Name, spec.Args, nil)
        }
    }
    // Fallback to go test.
    if goOld {
        fmt.Fprintf(os.Stderr, "hint: prism requires Go 1.24+; using go test (you are on %s)\n", runtime.Version())
    } else {
        fmt.Fprintln(os.Stderr, "hint: prism not found; install with: go install go.dalton.dog/prism@latest")
    }
    spec := ToolSpec{Name: "go", Args: []string{"test", "./..."}}
    return runTool(spec.Name, spec.Args, nil)
}

func goVersionLessThan(want string) bool {
    v := strings.TrimPrefix(runtime.Version(), "go")
    return v < want
}
```

The `goVersionLessThan` helper uses lexicographic string comparison on Go's semver, which is correct for "1.24" < "1.25" etc. (no need for a full semver parser).

**Verify:** Unit tests:
- `TestTest_PrismPreferred` (mocks `prism` via a tempdir on `$PATH`; verifies it's called with `go test ./...`)
- `TestTest_GoTestFallback` (runs in an env where `prism` is missing; verifies `go test ./...` is called with the install hint printed)
- `TestGoVersionLessThan` (table-driven: 1.23 < 1.24 = true, 1.24 < 1.24 = false, 1.25.0 < 1.24 = false)

The `prism` mocking strategy: prepend a tempdir to `$PATH` containing a `prism` shell script that echoes its args to a file. The test asserts the file's contents.

`fmt.go` per research §4.3 (gofumpt → goimports → gofmt chain, `--no-strict` bypasses gofumpt):
```go
func Fmt(noStrict bool) error {
    if path, err := exec.LookPath("gofumpt"); err == nil {
        if err := runTool(path, []string{"-l", "-w", "."}, nil); err != nil {
            return err
        }
    } else if !noStrict {
        return fmt.Errorf("gofumpt not found; install with: go install mvdan.cc/gofumpt@latest (or pass --no-strict)")
    } else {
        fmt.Fprintln(os.Stderr, "warn: gofumpt not found; falling back to gofmt (--no-strict)")
    }
    if path, err := exec.LookPath("goimports"); err == nil {
        if err := runTool(path, []string{"-w", "."}, nil); err != nil {
            return err
        }
    }
    return runTool("gofmt", []string{"-l", "-w", "."}, nil)
}
```

The `Fmt` function's signature takes `noStrict bool` so the caller (`cmd/fmt.go`) can pass the flag value through. The chain order is gofumpt → goimports → gofmt: gofumpt is a stricter gofmt (so gofmt after is idempotent), goimports adds/removes imports (gofumpt does not).

- `TestFmt_AllPresent` (mocks gofumpt + goimports + gofmt on `$PATH`; verifies all three are called in order)
- `TestFmt_GofumptMissing_Strict` (mocks only goimports + gofmt; verifies Fmt returns error mentioning gofumpt install hint)
- `TestFmt_GofumptMissing_NoStrict` (same mock; passes noStrict=true; verifies Fmt succeeds with a warn message to stderr)

## Anti-patterns

- Do NOT block the user on `prism` not being installed -- `prism` is the upgrade path, `go test` is the always-works fallback. The wrapper must always exit cleanly even on a barebones CI image.
- Do NOT use `exec.Command` directly without `cmd.Stdin = os.Stdin` + `cmd.Stdout = os.Stdout` + `cmd.Stderr = os.Stderr` -- the user expects to see the wrapped tool's output, not a silent wrapper.
- Do NOT add a `--strict` flag (the inverse of `--no-strict`) -- the default is strict, the flag is the opt-out. Naming convention follows `gofmt -l` (which uses the same pattern).
- Do NOT add a "find best tool" heuristic (e.g., preferring `golangci-lint` over `go vet`) -- the wrapper is a thin shim, the user picks the tool they want.
- Do NOT add a config file (`~/.spin.yaml`) for the wrappers -- the user passes flags or the env. Config-file support is Phase 4 (`WS-02` in v2 requirements).
- Do NOT modify `_base/Taskfile.yml.tmpl` in this plan -- the template already has the correct `setup:` target from Phase 1. The grep script validates it. If the template is wrong, the grep test catches it.
- Do NOT modify `_base/.air.toml.tmpl` in this plan -- the template already uses `build.entrypoint` from Phase 1. The grep script validates it.
