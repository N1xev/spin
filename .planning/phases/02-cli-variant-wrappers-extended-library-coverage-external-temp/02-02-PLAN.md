# Plan 02-02: External template override + FuncMap scaffolding for new libs

## Phase Goal

**As a** Go developer, **I want to** generate a working CLI or TUI project scaffolded with any charmbracelet v2 library, **so that** I can start building immediately without manual setup of dependencies, tooling, or build configs.

This plan ships the `--template-repo` override (TMPL-03) so users can pull a custom template tree from a git repo instead of using the embedded one, and it scaffolds the template engine with the FuncMap helpers Plan 02-03 needs to wire the 6 new lib overlays (`huh`, `glamour`, `glow`, `wish`, `log`, `harmonica`) into the generated go.mod and main.go.

The external template override is two new files (`fs.go` for the `templateFS` abstraction, `repo.go` for the git clone) plus a 5-line refactor to `template.go`'s walker to read from `currentFS(p.ExternalDir)` instead of the package-level `FS` var. The FuncMap additions are the new `hasX` helpers and the extended `charmPin` switch — they live in the same `template.go` file as the walker refactor, so they ship together.

## Context

Phase 1's template engine reads everything from the package-level `FS embed.FS` declared in `scaffold.go`. The walker in `template.go` does `fs.WalkDir(FS, root, ...)` and `fs.ReadFile(FS, path)` directly. To support an external template repo we need the walker to read from either the embedded FS (default) or `os.DirFS(<tempdir>)` (after a `git clone`). The smallest change is a 5-line `currentFS` swap — both `embed.FS` and `os.DirFS` already satisfy the `templateFS` interface (a 3-method subset of `fs.FS`), so the swap is type-safe at compile time.

Plan 01 of Phase 1 (research §15.11) explicitly deferred the external repo wiring to Phase 2: "Templates embedded via `go:embed`; `--template-repo` override wired in Phase 2." The deferred design said "5-line refactor to template.go plus a new `p.ExternalDir string` field on `Project`." This plan delivers exactly that, plus the URL validation and `git clone` lifecycle the deferred design omitted.

For the FuncMap additions, Phase 1 already has `hasCobra`, `hasFang`, `requiresImport` (forward-compat for Phase 2), and the 3 active TUI helpers (`hasBubbletea`, `hasBubbles`, `hasLipgloss`). Phase 2 needs `hasHuh`, `hasGlamour`, `hasGlow`, `hasWish`, `hasLog`, `hasHarmonica`, `hasViper` as top-level helpers so Plan 02-03 can write `{{if hasHuh .}}charm.land/huh/v2 v2.0.3{{end}}` in the go.mod template.

## Tasks

### Task 1: Add `templateFS` interface + `currentFS` helper in `internal/scaffold/fs.go`

**Files:** `internal/scaffold/fs.go` (new)

**Action:** Create the new file with the `templateFS` interface (3 methods: `fs.FS`, `fs.ReadDirFS`, `fs.ReadFileFS`) and the `currentFS(externalDir string) templateFS` constructor. When `externalDir == ""`, return the package-level `FS` (embed.FS). When `externalDir != ""`, return `os.DirFS(externalDir).(templateFS)`. The type assertion is a compile-time guard: if `os.DirFS` ever drops one of the three methods, the build fails here (better than a runtime nil-method panic).

The file's package comment documents that both `embed.FS` (since Go 1.16) and `os.DirFS` results satisfy `templateFS`, and explains why the type assertion is needed (Go's type system can't statically prove an interface satisfaction for a concrete type returned by a function).

**Verify:** `go build ./...` exits 0; the type assertion compiles (verifies both `FS` and `os.DirFS` results satisfy the interface). No runtime test for the constructor itself; the integration test in Task 6 covers the end-to-end behavior.

**Commit:** `feat(02): add templateFS interface + currentFS for external template override`

### Task 2: Add `CloneTemplateRepo` in `internal/scaffold/repo.go`

**Files:** `internal/scaffold/repo.go` (new)

**Action:** Implement the `CloneTemplateRepo(ctx context.Context, url string) (string, error)` function per research §5.1:

1. `os.MkdirTemp("", "spin-template-*")` to get a tempdir.
2. `exec.CommandContext(ctx, "git", "clone", "--depth", "1", url, tmp)` with env `GIT_TERMINAL_PROMPT=0` + the 4 GIT_AUTHOR_*/GIT_COMMITTER_* vars from `git.go` `gitEnv` (reuse the constant — refactor `gitEnv` from package-private to exported or copy it; copying is simpler and keeps the seam small).
3. On git exit non-zero: `os.RemoveAll(tmp)` and return a wrapped error that includes git's stderr.
4. After successful clone: `os.Stat(filepath.Join(tmp, "_base"))` — if missing, `os.RemoveAll(tmp)` and return error "template repo %s: missing _base/ directory (required by spin's overlay engine)".
5. Return the tempdir path; the caller is responsible for cleanup.

The function takes `ctx` for future cancellation support (per CLAUDE.md `context.Context` patterns). Phase 2's call site (`runNew`) passes `context.Background()` but the signature is future-proof.

**Verify:** `go build ./...` exits 0. The unit test in Task 5 (TDD: write first, expect RED) covers the `_base/` validation branch and the git-failure branch via a mocked `exec` (or a tiny test fixture repo on disk via `git init` + `git add`).

**Commit:** `feat(02): add CloneTemplateRepo for --template-repo`

### Task 3: Add `ExternalDir` and `KeepTemplateCache` fields to `Project`

**Files:** `internal/scaffold/project.go`

**Action:** Add two fields to the `Project` struct:

- `ExternalDir string` — the tempdir returned by `CloneTemplateRepo`, or empty if `--template-repo` was not passed. The `currentFS` helper reads this field.
- `KeepTemplateCache bool` — when true, the `runNew` defer that calls `os.RemoveAll(p.ExternalDir)` is skipped. Useful for debugging a broken template repo.

Both fields are unexported from the user's perspective but visible to the scaffolder. The struct comment must explain that these are pipeline-internal: callers populate them via `ResolveFlags` + `CloneTemplateRepo`, not directly.

**Verify:** `go build ./...` exits 0; the existing tests in `project.go` / `template_test.go` / `validate_test.go` still pass (no field is referenced yet by tests, so adding fields is additive).

**Commit:** `feat(02): add ExternalDir + KeepTemplateCache to Project`

### Task 4: Refactor `template.go` walker to use `currentFS` + extend `funcMap`

**Files:** `internal/scaffold/template.go`

**Action:** Two changes in the same file:

1. **Walker refactor (5 lines):** In `renderToMap()`, replace the two references to the package-level `FS` (the `fs.WalkDir(FS, root, ...)` call and the `fs.ReadFile(FS, path)` call) with calls that use a local `fsys := currentFS(p.ExternalDir)` value. The walker iterates overlay layers the same way; only the source FS changes.

2. **FuncMap additions:** Add 7 new helpers to the `funcMap(p *Project)` return value:

The walker refactor is a single-commit change. The FuncMap additions are a separate semantic change but live in the same file; combining them avoids two commits to the same function.

### Task 5: Add `--template-repo` and `--keep-template-cache` flags

**Files:** `cmd/new.go`, `internal/scaffold/resolve.go`, `internal/scaffold/resolve_test.go`

**Action:** Register the two flags in `cmd/new.go` `init()` (next to the existing template/license flags):

- `--template-repo`: string, default `""`, description: "override the embedded template with an external git repo (depth-1 clone to a tempdir; the repo must have a _base/ directory)"
- `--keep-template-cache`: bool, default `false`, description: "retain the cloned template repo on disk after scaffolding (useful for debugging external templates)"

Bind them in `resolve.go`: add `p.TemplateRepo` (string) and `p.KeepTemplateCache` (bool) reads. The `ExternalDir` field on `Project` is NOT bound from a flag — it is set by `runNew` after the clone succeeds (so the `git clone` error path can return without mutating the struct).

Then in `runNew` (which currently lives in `cmd/new.go`), add the clone logic between `p.Validate()` and `scaffold.New(p)`:

1. If `p.TemplateRepo != ""`:
   - Call `scaffold.CloneTemplateRepo(ctx, p.TemplateRepo)` (use `context.Background()` for now; future plans may plumb a `--ctx-timeout` flag).
   - On error, return the wrapped error.
   - Set `p.ExternalDir = dir`.
   - If `!p.KeepTemplateCache`, `defer os.RemoveAll(dir)`.
2. Log the tempdir path at Info level so users with `--keep-template-cache` can find it.

Also in `resolve.go`, add the URL format check: if `TemplateRepo != ""`, it must start with `http://`, `https://`, `git://`, or `git@` (for ssh-agent URLs). Return an `ArgError` if not. This is permissive — `git clone` is the actual choke point for invalid URLs.

**Verify:** `go test ./internal/scaffold/... -run TestResolveFlags_TemplateRepo` passes (added in Task 6) — assert `--template-repo https://example.com/foo` sets `p.TemplateRepo = "https://example.com/foo"` and `p.ExternalDir == ""` (set by runNew, not resolve). `go test ./internal/scaffold/... -run TestResolveFlags_KeepTemplateCache` passes — assert the bool binds.

**Commit:** `feat(02): wire --template-repo and --keep-template-cache flags`

## Anti-patterns

- Do NOT block on credentials in the git clone — `GIT_TERMINAL_PROMPT=0` is mandatory. Users with private repos use `GIT_SSH_COMMAND` or ssh-agent, not terminal prompts.
- Do NOT add an `eval` or `replace` function to the FuncMap — the path-traversal guard in Plan 02-01's `emit()` is the only line of defense; a template-side `replace` would let users render `{{.Name}}` to anything.
- Do NOT add a `--template-repo-cache-dir` flag — the tempdir is per-invocation and `os.RemoveAll`'d on completion. `--keep-template-cache` is the only escape hatch.
- Do NOT add per-lib variants of the funcMap helpers (`hasV2Huh`, `hasCharmlandHuh`) — the canonical helper is the simple `hasHuh` reading the bool field. Plan 02-03 references them as `{{if hasHuh .}}` and that's it.
- Do NOT touch `_base/go.mod.tmpl` to add 6 new lib require entries — that's Plan 02-03's job. This plan only adds the FuncMap helpers; the templates that use them ship next.
