## User Constraints (from CLAUDE.md and 03-UI-SPEC.md)

The UI-SPEC is the **locked design contract** for this phase. Items below are treated as locked decisions; research enforces them, not explores alternatives.

### Locked Decisions (UI-SPEC + CLAUDE.md)

1. **`gum` is a binary dep, not a Go library.** Shell out via `os/exec`. `charm.land/gum/v2` does not exist (gum is binary-only) — do not try to import.
2. **Fallback is `charm.land/huh/v2` v2.0.0+, used in-process.** No `survey`, no `promptui`, no `bubbletea` program wrapping the fallback.
3. **Backend selection is decided once per `spin new` invocation** and never switches mid-flow. Order: `exec.LookPath("gum")` → if found, gum; else if TTY, huh; else no prompts (fail fast like Phase 2).
4. **Three-layer prompt guard:** `SPIN_NO_INTERACTIVE` env → `isatty.IsTerminal(os.Stdin)` → CI env vars (`$CI`, `$GITHUB_ACTIONS`, `$GITLAB_CI`, `$BUILDKITE`, `$CIRCLECI`). All three must pass to enable prompting.
5. **`--no-interactive` / `--yes` / `--batch` are aliases for a single internal bool** (`p.NoInteractive`). One flag binding, three CLI spellings via cobra's `cmd.Flags().Lookup(...).Aliases` (cobra 1.8+).
6. **AGENTS.md is opt-in only.** Defaults to off. Default in prompt = `Yes` (dogfooding bias).
7. **AGENTS.md marker on line 1, no blank line above:** `<!-- AUTOGENERATED by spin {{.SpinVer}} -->`. Format is GFM, zero ANSI, no lipgloss.
8. **AGENTS.md is deterministic:** sections in fixed order, libs sorted alphabetically (re-uses `sort.Strings` from ResolveFlags), no timestamps, no UUIDs, no machine IDs. Only variable: `{{.SpinVer}}`.
9. **AGENTS.md MUST NOT contain** secrets, env values, user-specific paths, `.env` examples, ANSI/lipgloss styles, external URLs other than `charm.land/...` and `https://charm.land/`, "TODO" placeholders, commit hashes, or "generated by AI" copy.
10. **`Project` struct remains the single source of truth.** Prompt answers and flag values populate the same fields. The contract: `internal/prompt` exports exactly one function — `func Fill(p *scaffold.Project) error` — called from `cmd/new.go` between `ResolveFlags` and `p.Validate()`.
11. **Prompt sequence is fixed and named in UI-SPEC** (8 steps: type → name → module → libs → license → template → external repo → AI). Step order MUST match the spec.
12. **Copy strings are locked** in the UI-SPEC's "Copywriting Contract" table. Exact strings, no paraphrasing.
13. **Cobra `Args` must be relaxed** from `cobra.ExactArgs(1)` to allow the project-name prompt to fire. New rule: `cobra.MaximumNArgs(1)`, then check `len(args) == 0` in `runNew` and prompt for name when empty.
14. **Cancellation = `*prompt.Canceled` error, exit code 130.** No FS writes happen during prompts (prompts run before `scaffold.New`), so cleanup is just "exit cleanly with the right code". `main.go` handles the exit-code mapping.
15. **gum v0.14+ is the target.** The flags used (`choose`, `input`, `confirm`, `--no-limit`, `--header`, `--placeholder`, `--prompt`) are stable since v0.10. Latest at time of research: v0.17.0.
16. **huh v2 uses `huh.ErrUserAborted` (sentinel)** returned from `form.Run()` on Ctrl-C (mapped from `tea.ErrInterrupted`). Use `errors.Is(err, huh.ErrUserAborted)`.
17. **AGENTS.md is rendered through Go `text/template` like the rest of the scaffold.** No second rendering path.
18. **No new CLI subcommands in Phase 3** — the only new surface is `AGENTS.md` and the prompt flow on `spin new`.

## Summary

Phase 3 wires the missing 20% of the scaffolder: a TTY-aware prompt loop and an opt-in AI-context file. The core insight is that prompts and flags must feed the **same struct** (`scaffold.Project`), and the prompt loop must run **before** any filesystem write so cancellation costs nothing. The two surfaces — gum subprocess calls and huh v2 in-process forms — implement identical observable behavior so tests assert against the *Project after `Fill`, not the widget internals.

The AGENTS.md generation is a template-engine extension, not a separate code path. A new `lib/ai/` overlay directory holding `AGENTS.md.tmpl` is gated on the existing `p.AI` bool via the `boolFlagOverlayMap` pattern (the engine already supports Phase 2 libs this way). A new FuncMap helper `charmLibInfo` provides the per-library display name, module path, purpose, extending guidance, and example — all static text from a fixed lookup table, not user-supplied content. Determinism comes from: fixed section order, alphabetical lib sort, no timestamps, no UUIDs.

The single biggest design choice is the **backend resolver** — a one-shot `gumAvailable()` probe at the top of `Fill` that decides gum/huh/none and never switches. This is gated by `ShouldPrompt()`, which itself is gated by three layers (env var, TTY, CI). The `cmd/new.go` change is minimal: relax cobra's `Args` to `MaximumNArgs(1)`, call `prompt.Fill(p)` between `ResolveFlags` and `p.Validate()`, and add `os.Exit(130)` handling in `main.go` for `*prompt.Canceled`.

**Primary recommendation:** Build `internal/prompt` as a thin package with one entry point (`Fill`), two backends (`gum.go` shell-out, `huh.go` in-process), and a `ShouldPrompt()` chokepoint that all entry points go through. Add `lib/ai/AGENTS.md.tmpl` for the opt-in file. Wire `cmd/new.go` to call `Fill` between flag-binding and validation. Add a `*prompt.Canceled` error and a 3-line change in `main.go` to map it to exit code 130.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| TTY detection (`isatty.IsTerminal`) | CLI (scaffolder) | — | Pure runtime check on `os.Stdin`; no network/IO |
| CI env var detection | CLI (scaffolder) | — | Reads `$CI`/`$GITHUB_ACTIONS`/etc. from `os.Getenv` |
| `--no-interactive` flag binding | CLI (cobra layer) | — | Standard cobra flag wiring in `cmd/new.go init()` |
| gum binary probe (`exec.LookPath`) | CLI (scaffolder) | — | One-shot at start of `Fill`; result cached in package-local var |
| gum subprocess invocation | CLI (scaffolder) | — | `os/exec.CommandContext`; stdin/stdout wired to parent's tty |
| huh v2 in-process form | CLI (scaffolder) | — | `huh.NewForm(...).Run()`; uses bubbletea v2 underneath |
| TTY-detect helper exported | `internal/prompt` | — | Single chokepoint per INT-03 |
| Prompt result write-back to *Project | `internal/prompt` | — | Fills struct fields in place; no parallel state per INT-05 |
| AGENTS.md template (text/template) | `internal/scaffold` (existing) | `internal/prompt` (AI bool) | Reuses overlay engine; gated on `p.AI` via `boolFlagOverlayMap` |
| AGENTS.md library lookup table | `internal/scaffold` (FuncMap) | — | Static data; `charmLibInfo(name string) LibInfo` |
| Cancellation → exit code 130 | `main` boundary | `internal/prompt` (typed err) | `errors.Is(err, &prompt.Canceled{})` check in main; fang is just for styled output |
| File emit (writes `AGENTS.md`) | `internal/scaffold` (existing) | — | `scaffold.New(p)` walks overlay; `lib/ai/AGENTS.md.tmpl` becomes `AGENTS.md` |
| Post-scaffold smoke test | `internal/scaffold` (existing) | — | `go build ./...` runs as today; AGENTS.md is just another rendered file |

**Key takeaway:** Phase 3 has no new tiers. Everything happens in the CLI process, all writes go through the existing overlay engine, and the only new package is `internal/prompt` (no new tiers, just a new module-local concern).

### Pattern 1: TTY-aware backend resolver

**What:** One probe at the top of `Fill` selects the backend. The result is cached for the duration of the call. No mid-flow switching.

**When to use:** Whenever a tool has multiple implementations whose selection is environment-driven, not user-driven.

**Example:**

```go
// internal/prompt/prompt.go (sketch)
package prompt

import (
    "errors"
    "os/exec"

    "github.com/example/spin/internal/scaffold"
)

type backend int

const (
    backendNone backend = iota // no gum, no TTY → fail fast
    backendGum                  // gum binary on $PATH
    backendHuh                  // in-process huh v2
)

// resolveBackend decides once which prompt backend to use. The result
// is per-invocation; we never switch backends mid-flow. The check
// happens AFTER ShouldPrompt() returns true (i.e., TTY + not CI + not
// --no-interactive), so we know stdin is a real terminal.
func resolveBackend() backend {
    if path, err := exec.LookPath("gum"); err == nil {
        // Optional sanity-check: gum --version. A corrupt install
        // returns non-zero. We don't error out — we just fall through.
        if cmd := exec.Command(path, "--version"); cmd.Run() == nil {
            return backendGum
        }
    }
    return backendHuh // TTY is already verified by ShouldPrompt()
}

// Fill populates any unset required fields on p by asking the user.
// It writes back into p in place. If ShouldPrompt() returns false,
// Fill returns nil without modifying p and any missing fields will
// surface in the subsequent p.Validate() call.
//
// If the user cancels (Ctrl-C, Esc, empty-after-retry), Fill returns
// a *prompt.Canceled error; the caller exits 130.
func Fill(p *scaffold.Project) error {
    if !ShouldPrompt() {
        return nil
    }
    // If every required field is already populated, no need to prompt.
    if p.complete() {
        return nil
    }
    be := resolveBackend()
    switch be {
    case backendGum:
        return fillWithGum(p)
    case backendHuh:
        return fillWithHuh(p)
    default:
        // ShouldPrompt() returned true but resolveBackend() returned
        // backendNone — only happens if the TTY disappeared between
        // the two checks (e.g., user detached). Treat as cancel.
        return &Canceled{Reason: "no TTY available for prompts"}
    }
}

var ErrCanceled = errors.New("prompt canceled by user")

type Canceled struct{ Reason string }

func (c *Canceled) Error() string { return "spin: " + c.Reason }
func (c *Canceled) Is(target error) bool { return target == ErrCanceled }
```

### Pattern 2: gum subprocess with graceful cancellation

**What:** Each gum widget is one `os/exec` call. `cmd.Cancel` and `cmd.WaitDelay` are set so Ctrl-C kills gum promptly without leaving the scaffolder hanging.

**When to use:** Any subprocess that might receive Ctrl-C while attached to the parent's tty.

**Example:**

```go
// internal/prompt/gum.go (sketch)
package prompt

import (
    "bytes"
    "context"
    "fmt"
    "os"
    "os/exec"
    "strings"
    "time"
)

// gumRun runs a single gum widget and returns its stdout (trimmed).
// Args: e.g. []string{"choose", "--header", "X", "a", "b", "c"}.
func gumRun(args ...string) (string, error) {
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Minute)
    defer cancel()
    cmd := exec.CommandContext(ctx, "gum", args...)
    cmd.Stdin = os.Stdin
    cmd.Stdout = os.Stdout // gum writes its own UI
    var stderr bytes.Buffer
    cmd.Stderr = &stderr
    // CRITICAL: when ctx is canceled (Ctrl-C / 5min timeout), kill
    // gum and force pipes closed so we don't hang on Read. Same
    // pattern as scaffold/repo.go's git clone.
    cmd.Cancel = func() error {
        if cmd.Process == nil { return nil }
        return cmd.Process.Kill()
    }
    cmd.WaitDelay = 100 * time.Millisecond
    if err := cmd.Run(); err != nil {
        // gum exits 130 (or any non-zero) on Ctrl-C / Esc → cancel.
        if ctx.Err() == context.Canceled || ctx.Err() == context.DeadlineExceeded {
            return "", &Canceled{Reason: "gum " + args[0] + " canceled"}
        }
        return "", fmt.Errorf("gum %s: %w: %s", args[0], err, stderr.String())
    }
    return "", nil
}

func gumChoose(header string, options []string, defaultIdx int) (string, error) {
    args := []string{"choose", "--header", header, "--selected", fmt.Sprintf("%d", defaultIdx+1)}
    args = append(args, options...)
    // ... but we want stdout to be the answer, not the UI. So we need
    // a different wiring: gum writes UI to /dev/tty, the answer to stdout.
    // Solution: use --output=/dev/stdout (the default), but redirect
    // gum's UI to /dev/tty via the TTY env var, or accept that gum
    // writes the answer to stdout AND the UI to stdout interleaved.
    // ACTUALLY: gum's behavior is "write the chosen value to stdout
    // only; UI goes to the tty internally". We just need to wire
    // cmd.Stdout to a bytes.Buffer and print the result ourselves.
    // ... (see pattern below for the actual wiring)
}
```

**The right gum stdout pattern** (corrected from above): gum's widgets print the **chosen value to stdout** and the TUI-rendered UI to `/dev/tty` (or stderr depending on the build). The cleanest way to capture the value is:

```go
func gumRunCapture(args ...string) (string, error) {
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Minute)
    defer cancel()
    cmd := exec.CommandContext(ctx, "gum", args...)
    cmd.Stdin = os.Stdin
    var stdout, stderr bytes.Buffer
    cmd.Stdout = &stdout
    cmd.Stderr = &stderr
    cmd.Cancel = func() error {
        if cmd.Process != nil { return cmd.Process.Kill() }
        return nil
    }
    cmd.WaitDelay = 100 * time.Millisecond
    if err := cmd.Run(); err != nil {
        if ctx.Err() != nil {
            return "", &Canceled{Reason: "gum canceled"}
        }
        return "", fmt.Errorf("gum: %w: %s", err, stderr.String())
    }
    return strings.TrimRight(stdout.String(), "\n"), nil
}
```

Note: gum's `choose`, `input`, `confirm` widgets write the **answer to stdout only** — no UI noise mixed in. The UI is rendered via terminal control sequences sent to `/dev/tty` directly. This is the documented gum behavior (verified in gum README via Context7).

### Pattern 3: huh v2 in-process form with cancel detection

**What:** Build a `huh.Form` with one or more fields; call `form.Run()`. On `huh.ErrUserAborted` (mapped from `tea.ErrInterrupted`), return `*prompt.Canceled`.

**When to use:** Whenever in-process interactive form behavior is needed.

**Example:**

```go
// internal/prompt/huh.go (sketch)
package prompt

import (
    "errors"
    "fmt"

    "charm.land/huh/v2"

    "github.com/example/spin/internal/scaffold"
)

func fillWithHuh(p *scaffold.Project) error {
    // Step 1: project type (only if not already set by a flag).
    if p.Type == "" || !flagPassed("type") {
        var t string
        err := huh.NewSelect[string]().
            Title("What kind of project?").
            Options(
                huh.NewOption("TUI — terminal app with bubbletea", "tui"),
                huh.NewOption("CLI — command-line tool with cobra + fang", "cli"),
                huh.NewOption("TUI + CLI — single binary with both halves", "all"),
            ).
            Value(&t).
            Run()
        if errors.Is(err, huh.ErrUserAborted) {
            return &Canceled{Reason: "user aborted type selection"}
        }
        if err != nil {
            return fmt.Errorf("type prompt: %w", err)
        }
        p.Type = t
    }

    // Steps 2-7: similar single-field .Run() calls. Each one returns
    // huh.ErrUserAborted on Ctrl-C, which we map to *prompt.Canceled.
    //
    // For the library multi-select (step 4), use huh.NewMultiSelect[string]
    // with .Limit(0) for unlimited. Pre-populate .Selected(true) for
    // libs that the variant auto-enabled.
    //
    // For step 8 (AI opt-in), use huh.NewConfirm() with default true.

    return nil
}
```

**Critical huh v2 behavior** (verified via source at `form.go`):
- `Form.Run()` returns `huh.ErrUserAborted` on Ctrl-C (defined as `errors.New("user aborted")` at the top of `form.go`).
- `Form.Run()` does **no TTY check itself** — it delegates to `tea.NewProgram`. On a non-TTY stdin, the underlying bubbletea program will fail with an error like "could not open a new TTY" (this is why `ShouldPrompt()` is mandatory before calling `fillWithHuh`).
- The default keymap binds Ctrl-C to `keymap.Quit` (the abort path). Esc is delegated to fields, not bound to abort by default.
- The form's `CancelCmd` is `tea.Interrupt`; the form checks `errors.Is(err, tea.ErrInterrupted)` after Run to decide whether to return `ErrUserAborted`.

### Anti-Patterns to Avoid

- **Anti-pattern: prompt inside `scaffold.New`.** The scaffolder must not invoke prompts after FS writes begin. All prompting happens before `scaffold.New`.
- **Anti-pattern: separate `*PromptAnswers` struct.** Do not create a parallel state container. The whole point of INT-05 is one source of truth (`*Project`); the prompt code mutates `*Project` in place.
- **Anti-pattern: switch backends mid-flow.** If the user starts with gum and then the gum binary disappears (it won't, but defensively), do not fall through to huh. Commit to one backend per invocation.
- **Anti-pattern: prompt for every field, even when set.** `Fill` must skip any field already populated by a flag. Use `cmd.Flags().Changed(name)` or check `p.<field> != ""` (depending on the field's zero value semantics).
- **Anti-pattern: wrap huh v2 in our own bubbletea program.** Adds 50+ lines of glue for no observable behavior change. huh already integrates with bubbletea v2 underneath.
- **Anti-pattern: import `github.com/charmbracelet/gum` as a Go module.** It is binary-only. There is no Go library to import. Use `os/exec` and call the binary.

## Common Pitfalls

### Pitfall 1: cobra `ExactArgs(1)` blocks the project-name prompt

**What goes wrong:** With `cobra.ExactArgs(1)`, `spin new` (no args) fails at the cobra layer with `accepts 1 arg(s), received 0` before `runNew` is even called. The user cannot trigger the project-name prompt.

**Why it happens:** Phase 1/2 hardcoded `cobra.ExactArgs(1)` because the name was always required as a positional arg. Phase 3 wants to make it optional when `ShouldPrompt()` is true.

**How to avoid:** Change `newCmd.Args` to `cobra.MaximumNArgs(1)`. In `runNew`, if `len(args) == 0 && ShouldPrompt()`, the project-name prompt fires. If `len(args) == 0 && !ShouldPrompt()`, return the existing `ArgError{Message: "missing project name (positional argument)"}` (preserves the Phase 2 CI contract).

**Warning signs:** `spin new` with no args still errors out before the prompt code is reached; tests that expect the prompt to fire on missing name never see the prompt.

### Pitfall 2: `huh.ErrUserAborted` is missed

**What goes wrong:** Treating any non-nil error from `form.Run()` as a fatal error means a Ctrl-C exit shows a stack trace or fang-styled error instead of clean "prompt canceled" + exit 130.

**Why it happens:** huh v2 returns `huh.ErrUserAborted` (a sentinel `errors.New("user aborted")` defined at the top of `form.go`) on Ctrl-C. This is not a special "Cancel" method — it's just a sentinel you check with `errors.Is`.

**How to avoid:** In every place that calls `form.Run()` (or a single-field `.Run()`), the error check is:

```go
if errors.Is(err, huh.ErrUserAborted) {
    return &prompt.Canceled{Reason: "..."}
}
if err != nil {
    return fmt.Errorf("...: %w", err)
}
```

**Warning signs:** Ctrl-C during a prompt shows "exit status 1" with a stack trace; the "prompt canceled" message never appears.

### Pitfall 3: gum installed but broken → no fallback

**What goes wrong:** `exec.LookPath("gum")` finds the binary, but `gum --version` fails (corrupt install, missing shared lib, etc.). The first `gum choose` call hangs or exits non-zero, and there's no fallback to huh.

**Why it happens:** We commit to one backend at the top of `Fill`; once we pick `backendGum`, we never switch.

**How to avoid:** Two layers of defense:
1. At `resolveBackend()` time, run `gum --version` to sanity-check the install. If it fails, downgrade to `backendHuh`.
2. At each `gumRun` call, treat a non-zero exit on the first call as "gum broken" and bubble that up to the user with a clear "gum is installed but broken; rebuild with huh fallback" message. (The alternative is to swap backends mid-flow, which we explicitly do not do.)

**Warning signs:** `gum` exists on `$PATH` but the first prompt hangs or exits 1; the user sees no huh fallback.

### Pitfall 4: `p.Libs` and the Phase 2 bools drift

**What goes wrong:** The user adds `--huh` via a flag (so `p.Huh = true` but `p.Libs` does NOT contain `"huh"`). The prompt sees "huh is set" and offers it as pre-selected. The user un-selects it. The prompt code sets `p.Huh = false` but doesn't touch `p.Libs`. Everything still works — except the AGENTS.md template iterates over `p.Libs` only (or only the bools, depending on the helper), so the rendering misses a lib.

**Why it happens:** The current `ResolveFlags` and `boolFlagOverlayMap` treat `p.Libs` and the bools as **two parallel sources of truth**. The overlay engine walks both, but the AGENTS.md template will iterate over only one (recommended: iterate over `p.Libs` after re-deriving it from the bools, or iterate over the bools directly).

**How to avoid:** Define a single helper in `internal/scaffold`:

```go
// AllLibs returns the unified library set: every entry in p.Libs plus
// every bool set in boolFlagOverlayMap, deduplicated, sorted alphabetically.
func (p *Project) AllLibs() []string {
    seen := map[string]bool{}
    var out []string
    for _, lib := range p.Libs {
        if !seen[lib] { seen[lib] = true; out = append(out, lib) }
    }
    for lib, active := range p.boolFlagOverlayMap() {
        if active && !seen[lib] { seen[lib] = true; out = append(out, lib) }
    }
    sort.Strings(out)
    return out
}
```

Use `p.AllLibs()` everywhere a "complete library list" is needed: the AGENTS.md template, the prompt default-selection, the overlay walker's order. The planner should consider refactoring `overlayOrder` to call `p.AllLibs()` as well (low-risk, but worth flagging).

**Warning signs:** A user with `--huh` (and no `--huh` in `p.Libs`) gets an AGENTS.md that omits the Huh section; or vice versa.

### Pitfall 5: AGENTS.md template is non-deterministic

**What goes wrong:** Using `{{range .Libs}}` directly works only if `p.Libs` is sorted. If a future change to `ResolveFlags` breaks the sort, AGENTS.md becomes non-deterministic and PR diffs show "spurious" changes.

**Why it happens:** Easy to assume the input is sorted when it isn't.

**How to avoid:** In the AGENTS.md template, do not iterate `p.Libs` directly. Use a new FuncMap helper:

```go
"libsSorted": func(p2 *Project) []string {
    libs := append([]string{}, p2.Libs...)
    sort.Strings(libs)
    return libs
},
```

Or (better): make `p.AllLibs()` the source, which is always sorted (as shown in Pitfall 4 fix). Use `{{range allLibs .}}` in the template.

**Warning signs:** Two consecutive `spin new` invocations with the same flags produce byte-different AGENTS.md files.

### Pitfall 6: `p.Libs` vs prompt answer — type mismatch on multi-select

**What goes wrong:** The gum multi-select returns `["bubbletea", "bubbles", "lipgloss"]` (string slice). The huh multi-select returns the same. But the existing `p.Libs` is `[]string` and the bools are `bool`. The prompt code needs to translate multi-select answers into BOTH list and bool updates.

**Why it happens:** Two parallel sources of truth (see Pitfall 4).

**How to avoid:** Use `p.AllLibs()` to derive the current state, let the prompt return a modified `[]string`, then write back to BOTH `p.Libs` (after sort+dedup) and the corresponding bool fields (`p.Cobra`, `p.Fang`, `p.Huh`, etc.) — OR refactor to use `p.AllLibs()` as the only source and compute bools/lists on the fly (larger refactor; defer to Phase 4 if not in scope).

**Warning signs:** After a multi-select prompt, the generated project is missing libraries the user explicitly selected; or the bools are out of sync with `p.Libs`.

## Open Questions

1. **Should the AGENTS.md include CLI subcommand documentation?**
   - What we know: UI-SPEC defines sections "What this project is", "Libraries", "Extending", "Conventions", "Rebuilding this file". The "Extending" section already covers "how to add a new command".
   - What's unclear: Whether to also include a per-subcommand list (e.g., "your CLI has these subcommands: foo, bar, baz"). This is data we don't have access to from `*Project` alone; would require either template-time rendering of `main.go` (fragile) or a separate template that pulls from a config (over-engineering for v1).
   - Recommendation: **Skip for v1.** The "Extending" section already tells the user where to add commands. If users want per-subcommand docs, that's a Phase 4+ feature.

2. **What happens if the user passes `--no-interactive` AND a flag has a value that requires confirmation?**
   - What we know: `--no-interactive` is the chokepoint; `ShouldPrompt()` returns false. No prompts fire. If `--license mt` (typo) is passed, `p.Validate()` catches it. If no `--tui`/`--cli`/`--all` is passed, `ResolveFlags` defaults to `--tui` (per existing code in resolve.go:131-141). So a `spin new myapp --no-interactive` invocation defaults to TUI.
   - What's unclear: Whether users expect `--no-interactive` to FAIL when no variant is passed, or to silently default to TUI.
   - Recommendation: **Default to TUI silently** (matches existing behavior). The UI-SPEC error table line "User passes `--no-interactive` AND a required flag is missing" applies to the case where the user explicitly wants non-interactive AND wants a specific variant they didn't pass. Document this in `spin new --help`.

3. **Should the prompt code handle the `--template` selection differently for `--all`?**
   - What we know: For `--tui`, default template is `tui-bubbletea`. For `--cli`, default is `cli-cobra-fang`. For `--all`, both are valid; user picks one.
   - What's unclear: The "all" variant has its own `variant_all/main.go.tmpl` but no specific template name (it's not a per-template variant).
   - Recommendation: **For `--all`, default the template prompt to `tui-bubbletea`** (matches the most common combined use case). If the user picks `cli-cobra-fang`, the overlay walker emits the CLI-specific files but the variant_all template still wraps both halves. The planner should verify this with the existing `variant_all/main.go.tmpl`.

4. **What about `SPIN_NO_INTERACTIVE` env var vs. `--no-interactive` flag precedence?**
   - What we know: UI-SPEC shows the env var as the first check in `ShouldPrompt()`. Both result in the same `false` return.
   - What's unclear: Should an env var `--no-interactive=false` override the flag? UI-SPEC does not specify.
   - Recommendation: **Env var wins.** If `SPIN_NO_INTERACTIVE` is set (any non-empty value), prompts are disabled. The flag is for users who don't want to set an env var. No "unset the env var" path.

5. **Should the prompt re-prompt on empty input, or fail immediately?**
   - What we know: UI-SPEC error table says "re-prompts once; second empty → exits 1 with `spin: <field> is required`".
   - What's unclear: What about for the `gum input` widget itself — does empty + Enter result in an empty string, or does gum treat it as "cancel"?
   - Recommendation: **Test empirically during implementation.** If gum treats empty + Enter as "submit empty string", our code catches it and re-prompts. If gum treats it as cancel, our existing `Canceled` mapping handles it. Either way, the user gets clear behavior.

### Applicable ASVS Categories

|---------------|---------|-----------------|

| V6 Cryptography | no | — (no secret material handled) |

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| User types a malicious `gum input` answer (e.g., path with `..`) | Tampering / EoP | `p.Validate()` already enforces name regex; `emit()` has path-traversal guard on rendered files. The prompt code just writes back to `*Project`; validation runs before any FS write. |
| User passes `--template-repo https://x.com/-evil` (CR-004 in Phase 2) | Tampering | `IsValidTemplateRepo` already rejects URLs with leading-dash paths. The prompt must call `IsValidTemplateRepo` on the input and re-prompt on failure. |
| Path-traversal in `AGENTS.md` template | Tampering | The template does not interpolate `{{.Name}}` into a path; it only goes into Markdown text. The `emit()` path-traversal guard in scaffold.go is a defense-in-depth. |
| `gum` binary is maliciously replaced on $PATH | Spoofing | Out of scope; user controls $PATH. The install hint is `go install github.com/charmbracelet/gum@latest`; the version sanity check (`gum --version`) detects a missing/corrupt install. |
| huh v2 dependency has a known CVE | Repudiation | Standard `go get` + `go mod tidy` updates; dependabot / renovate. Out of scope for this phase. |
| AGENTS.md leaks secrets | Information Disclosure | AGENTS.md is a static template; no secret material is interpolated. The template MUST NOT reference `{{.Module}}` in a way that includes user-specific paths (it does, but only as the module name, which the user controls). The "What AGENTS.md MUST NOT contain" list in UI-SPEC is the contract. |

**Key insight:** Phase 3 introduces one new attack surface — the prompt inputs — but all of them flow through the existing `p.Validate()` guard, so no new threat models are introduced. The security review of Phase 3 is essentially: "do the prompts write to *Project fields that p.Validate() then checks?". The answer is yes, so no new controls are needed.
