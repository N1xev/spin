# Phase 3 — UI Design Contract

> Visual and interaction contract for the two surfaces Phase 3 adds:
> (A) the interactive scaffolder prompt sequence (`gum` + `huh v2` fallback)
> and (B) the generated `AGENTS.md` artifact. Generated by gsd-ui-researcher,
> verified by gsd-ui-checker.

This phase ships a Go CLI scaffolder, not a frontend app. The shadcn-gate
does not apply: the project has no React/Next.js/Vite toolchain. The
"design system" is therefore the charm v2 terminal-stack + GitHub-flavored
Markdown, and the spacing/typography/color sections below are interpreted
through that lens (terminal columns, text roles, ANSI palette).

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Go CLI + embedded templates) |
| Preset | not applicable |
| Component library | charm.land/huh/v2 (in-process fallback forms) |
| Icon library | not applicable (TUI uses ASCII arrows + gum glyphs) |
| Font | system terminal (no font hint) |
| Scaffolder UI runtime | gum binary (charmbracelet/charm) on $PATH, fallback to huh v2 |
| Generated artifact format | GitHub-flavored Markdown (no ANSI, no lipgloss) |
| Reference renderer (scaffolder) | gum v0.14+ (styled TUI) |
| Reference renderer (fallback) | huh v2 (charm.land/huh/v2) |
| Reference renderer (AGENTS.md) | Markdown viewers (GitHub, VS Code, Claude Code, Cursor, Aider) |

## Spacing Scale (terminal columns / lines)

Declared values (column widths are always multiples of 4 where possible):

| Token | Columns | Usage |
|-------|---------|-------|
| xs | 1 col | cursor / focus gap |
| sm | 2 col | inline gap between icon and label |
| md | 4 col | default element spacing (matches "default gap" in gum) |
| lg | 8 col | section padding (gum `padding="0 2"` = 2 cols each side) |
| xl | 16 col | major section break (gum `padding="2 4"`) |
| 2xl | 32 col | page-level (reserved; not used in prompts) |
| 3xl | 64 col | page-level (reserved; not used in prompts) |

Exceptions:
- gum `choose` widget renders at 60 cols wide by default — we keep the default (no override).
- gum `input` renders at 40 cols wide — we keep the default.
- Prompt question text may wrap to 80 cols; gum handles this internally.

## Color (ANSI palette)

The scaffolder uses charm/log v2 for stderr and gum's default palette for
prompts. No custom colors — gum's defaults are the design.

| Role | Palette | Usage |
|------|---------|-------|
| Dominant (60%) | terminal default fg + bg | gum prompt background, scaffold stdout |
| Secondary (30%) | terminal default fg (no styling) | hint text, secondary labels |
| Accent (10%) | gum default selection (cyan/magenta) | focused choice in `gum choose`, focused field in `gum input`, focused button in `gum confirm` |
| Destructive | ANSI red (`\033[31m`) | `--no-interactive` violation errors, gum-missing install hint, non-TTY errors |

Accent is reserved for: (1) the currently-focused item in any prompt, (2)
the success log line `Created at ./<name>`. Nothing else.

AGENTS.md MUST contain zero ANSI escape codes, zero lipgloss styles, zero
hex colors. It renders as plain GitHub-flavored Markdown.

## Surface A: Scaffolder Prompts

### Trigger logic

Prompts only fire when ALL of the following are true:

1. `cmd.Name() == "new"` and at least one required field is unset.
2. `isatty.IsTerminal(os.Stdin)` returns true (INT-03).
3. The user has NOT passed `--no-interactive` / `--yes` / `--batch` (INT-02).
4. We are not running inside a CI environment (`$CI`, `$GITHUB_ACTIONS`,
   `$GITLAB_CI`, `$BUILDKITE`, `$CIRCLECI` all unset) — CI is treated as
   non-TTY even if stdin is a tty (defense in depth, INT-03).

If any condition fails, prompts are skipped and any missing required field
returns the existing `ArgError` / `FlagError` (preserves the Phase 2 contract
that `spin new myapp` in CI fails fast with a clear message).

### Prompt sequence

The order is fixed. Each prompt is independent — a user can Ctrl-C / Esc at
any point and `spin` exits 130 (per Phase 2 conventions) with no FS writes.

| # | Field | Widget | Default | When skipped |
|---|-------|--------|---------|--------------|
| 1 | Project type | `gum choose` | `TUI` (1st option) | `--tui` / `--cli` / `--all` already set |
| 2 | Project name (only if args empty in `cobra.Args`) | `gum input` | empty | positional arg already provided |
| 3 | Module path | `gum input` | pre-filled with `<name>` | `--module` already set |
| 4 | Library multi-select | `gum choose --no-limit` | pre-checked: variant defaults | none — always asked, but defaults match variant |
| 5 | License | `gum choose` | `MIT` | `--license` already set |
| 6 | Template | `gum choose` | `tui-bubbletea` (or the matching variant default) | `--template` already set |
| 7 | External template repo | `gum input --placeholder="(skip to use embedded)"` | empty | `--template-repo` already set |
| 8 | AI opt-in (`AGENTS.md`) | `gum confirm` | `Yes` (matches dogfooding) | `--ai` / `--no-ai` already set, or `--no-interactive` |

Notes on the sequence:
- Steps 4 and 8 are the only steps that always fire (with the type-specific
  defaults pre-applied). Everything else is a true "fill the gap" prompt.
- Steps 2 and 7 are conditional on the user not having passed the value
  positionally / via flag.
- After step 8, a one-line summary prints to stderr (charm/log v2 `Info`):
  `Will scaffold <name> (<type>) with: <lib list>` then the actual scaffold
  runs.

### Widget choice rule

| Need | Widget | Why |
|------|--------|-----|
| Pick 1 of N | `gum choose` | gum's first-class single-select; huh equivalent: `huh.NewSelect` |
| Pick 0+ of N | `gum choose --no-limit` | gum's multi-select; huh equivalent: `huh.NewMultiSelect` |
| Free-form text | `gum input` | gum's single-line input; huh equivalent: `huh.NewInput` |
| Yes / no | `gum confirm` | gum's two-button confirm; huh equivalent: `huh.NewConfirm` |

### gum vs huh decision (INT-04)

The scaffolder probes for `gum` on `$PATH` at startup. Resolution order:

1. If `gum` is found via `exec.LookPath("gum")` -> use gum subprocess path.
2. Else if `gum` is not found AND we are in TTY -> use huh v2 in-process.
3. Else (non-TTY) -> no prompts; missing flags fail fast as in Phase 2.

The choice is decided once per `spin new` invocation and never mid-flow
(cannot switch backends). The decision is logged at Debug level:
`"prompt backend" backend=gum` or `backend=huh`.

### TTY guard (INT-03)

Every TUI entry point is wrapped by a single helper:

```go
// internal/prompt/prompt.go (new package, Phase 3)
func ShouldPrompt() bool {
    if os.Getenv("SPIN_NO_INTERACTIVE") != "" { return false }
    if isatty.IsTerminal(os.Stdin) == false { return false }
    if ciEnv() { return false }
    return true
}
```

The `isatty` import is `github.com/mattn/go-isatty` (or `charm.land/x/isatty`
if available at scaffold time). `ciEnv()` checks the CI env var list
above. This helper is the single chokepoint; no `gum.*` call site is allowed
to invoke gum without first calling `ShouldPrompt()`.

### Cancellation / cleanup

Ctrl-C / Esc in gum exits with code 130. The scaffolder wraps the prompt
loop in a `signal.Notify` handler that:

1. Logs the interrupt at Info: `prompt canceled by user`.
2. Calls `os.RemoveAll(p.ExternalDir)` if it was cloned (mirrors the
   `--template-repo` cleanup already in `cmd/new.go`).
3. Exits with `os.Exit(130)`.

No partial state can leak because prompts run BEFORE `scaffold.New` (which
is the only code that writes to `./<name>/`).

### Error states

| Condition | Message (stderr) | Exit code |
|-----------|------------------|-----------|
| Non-TTY + missing required flag | `spin: --tui or --cli is required (or run interactively in a terminal)` | 1 |
| `gum` not found AND huh v2 import is wired | `spin: gum not found on $PATH; falling back to in-process huh prompts` (Info, not Error) | 0 |
| `gum` not found AND huh v2 failed to import (build-time) | `spin: gum is not installed. Install it: go install github.com/charmbracelet/gum@latest` | 1 |
| `gum` exits non-zero (broken pipe, SIGPIPE) | `spin: prompt interrupted` | 130 |
| User passes `--no-interactive` AND a required flag is missing | `spin: --tui or --cli is required (--no-interactive disables prompts)` | 1 |
| User types empty `gum input` for required field | re-prompts once; second empty -> exits 1 with `spin: <field> is required` | 1 |
| `--template-repo` answered in prompt fails to clone | re-prompts once with the gum error; second failure -> exits 1 with clone error | 1 |

### Install hint (gum missing)

Single line, copy-pasteable, charm install:

```
spin: gum is not installed. Install it: go install github.com/charmbracelet/gum@latest
```

This message is also surfaced in `spin doctor` (Phase 4) but is emitted
inline in Phase 3 when the fallback path itself fails.

## Surface B: AGENTS.md Template

### File location

`./<name>/AGENTS.md` at the project root, next to `README.md` and `go.mod`.
Created only when the user opts in via `--ai` / `--agents` or answers Yes to
prompt #8. Skipped entirely otherwise (AI-04).

### Determinism contract

For a given set of `--flags` and a `spin` version, the rendered `AGENTS.md`
is byte-identical. This means:

- Sections are emitted in a fixed order (no map iteration).
- Library sections are emitted in alphabetical order (matches the existing
  `p.Libs` sort in `ResolveFlags`).
- No timestamps. No UUIDs. No machine IDs.
- The version string is the only variable: `<!-- AUTOGENERATED by spin {{.SpinVer}} -->`.

This makes the file safe to commit and easy to review in PRs.

### Marker (AI-02)

The first line of the file is an HTML comment marker:

```markdown
<!-- AUTOGENERATED by spin {{.SpinVer}} -->
```

It is the very first line (line 1, no blank line above). The marker is the
"owned-by-spin" signal: it tells `spin update` (Phase 4) and humans that
the file is auto-generated and may be overwritten by a future `spin new` of
the same flags.

### File structure (top to bottom)

| Section | Required | Content source |
|---------|----------|----------------|
| Marker | always | `<!-- AUTOGENERATED by spin {{.SpinVer}} -->` |
| `# AGENTS.md` | always | H1 title |
| `## What this project is` | always | one paragraph: `<Name>` is a `<Type>` Go project generated by `spin` on `<Year>`. Module path: `<Module>`. |
| `## Libraries` | always | one `### <lib>` block per library in `p.Libs` (alphabetical) |
| `## Extending` | always | how to add a new library, how to add a new command, where templates live |
| `## Conventions` | always | short bullet list (no CGO, charm v2 paths only, no v1 imports) |
| `## Rebuilding this file` | always | one line: `Regenerate with: spin new <name> <flags>` |

### Library block format (per `p.Libs[i]`)

```markdown
### <lib display name>

- Module: `charm.land/<lib>/v2` (or `github.com/<org>/<lib>` for non-charm)
- Purpose: <one-line description from a fixed lookup table>
- Extending: <2-4 sentence guidance specific to the library>
- Example: <one fenced code block, minimum 5 lines, copy-pasteable>
```

The fixed lookup table for charm v2 libraries is in the template, not in
generated code. Each entry has exactly: `display name`, `module path`,
`one-line purpose`, `extending guidance`, `example code`. The example is
rendered through Go's `text/template` like the rest of the scaffold.

|-----------------|--------------|--------|---------|---------------------|--------------|

### What AGENTS.md MUST NOT contain

- No secrets, no env values, no API keys, no paths to user-specific dirs
  (e.g. `/home/alice/...`). Use `<name>` / `<module>` placeholders.
- No `.env` examples unless the project actually uses them.
- No lipgloss / ANSI / color escape codes. Plain GFM.
- No external URLs except `charm.land/<lib>` module paths and the canonical
  charmbracelet docs root (`https://charm.land/`).
- No "TODO" / "FIXME" placeholders. The file is a finished doc.
- No copy that says "generated by AI". The marker is the only machine signal.
- No commit hash, build hash, or git remote URL.

### AI consumer expectations

AGENTS.md is consumed by Claude Code, Cursor, Aider, Continue.dev, and
similar. The convention those tools recognize (per their docs) is:
- File named exactly `AGENTS.md` (case-sensitive, no underscore).
- Located at the project root (not in a subdirectory).
- Uses standard GFM; tools parse with the same engine as README.md.

We follow that convention exactly so the file is auto-discovered.

## State Management (the single Project struct)

INT-05 is the load-bearing contract: there is exactly ONE source of truth,
and prompt answers feed the same struct that flag values populate.

### Flow

```
CLI flags ──┐
            ├──> cobra cmd ──> ResolveFlags ──> *Project (partial)
prompts ────┘                                       │
            (when ShouldPrompt() and gaps exist)    ▼
                                              *Project (complete)
                                                    │
                                                    ▼
                                              p.Validate()
                                                    │
                                                    ▼
                                              scaffold.New(p)
```

### Key invariant

A single package `internal/prompt` (new in Phase 3) owns all prompting.
It exports exactly one function with this signature:

```go
package prompt

// Fill populates any unset required fields on p by asking the user.
// It writes back into p in place. If ShouldPrompt() returns false,
// Fill returns nil without modifying p and any missing fields will
// surface in the subsequent p.Validate() call.
//
// If the user cancels (Ctrl-C, Esc, empty-after-retry), Fill returns
// a *prompt.Canceled error; the caller exits 130.
func Fill(p *scaffold.Project) error
```

There is no other entry point. Tests construct a *Project, call `Fill`,
and assert the result. `cmd/new.go` is the only production caller.

### Gating

`Fill` is a no-op when:

- `ShouldPrompt()` returns false (non-TTY, CI, `--no-interactive`).
- The user already supplied every required field via flags (the most
  common case: `spin new myapp --tui --bubbletea --bubbles`).
- The user supplied every required field via prompts on a prior step in
  the same invocation (does not re-ask).

The gating is checked at the top of `Fill`, not per-prompt, to keep
behavior predictable.

### Backends

`Fill` dispatches to one of two implementations based on the gum
availability probe:

| Backend | File | How it implements Fill |
|---------|------|-------------------------|
| gum | `internal/prompt/gum.go` | shells out to `gum choose` / `gum input` / `gum confirm` via `os/exec`; captures stdout; assigns back to `p` fields |
| huh | `internal/prompt/huh.go` | builds a `charm.land/huh/v2` form with one field per gap; calls `form.Run()`; assigns results to `p` fields |

Both implementations have the SAME observable behavior: same questions,
same defaults, same cancellation semantics, same write-back to `*Project`.
Tests assert against the `*Project` after `Fill`, not against the widget
internals.

### Field-by-field writeback (gum path)

| Prompt | Project field(s) written |
|--------|--------------------------|
| Project type | `p.Type` (set to `"tui"` / `"cli"` / `"all"`) |
| Project name | `p.Name` (only if empty; positional arg took precedence) |
| Module path | `p.Module` (only if empty) |
| Libraries | appends to `p.Libs`, then re-runs the sort+dedup pass (mirrors `ResolveFlags`) |
| License | `p.License` (lowercased, like `ResolveFlags`) |
| Template | `p.Template` |
| External template repo | `p.TemplateRepo` (validated by `IsValidTemplateRepo` after input) |
| AI opt-in | `p.AI` (true if user confirmed Yes) |

After `Fill` returns, `cmd/new.go` continues with `p.Validate()` and
`scaffold.New(p)` exactly as it does today. No branch on "did the
user prompt" downstream of `Fill`.

## Edge Cases & Invariants

1. **Flag-vs-prompt precedence:** flags always win. If a user passes
   `--tui` and also answers the project-type prompt (somehow), the flag
   is the truth. `Fill` skips any prompt whose target field is already
   non-zero / non-default.

2. **No double-prompt:** `Fill` is called exactly once per `spin new`
   invocation, in `runNew` after `ResolveFlags` and before `Validate`.

3. **No prompt during `spin run` / `spin test` / etc.:** only `spin new`
   calls `Fill`. The other subcommands do not prompt.

4. **AGENTS.md is regenerable:** the same flags produce the same file
   (byte-identical, per the determinism contract above). A user can
   `rm AGENTS.md && spin new . --ai --reuse-existing-flags` (future
   Phase 4) to regenerate without touching other files.

5. **`gum` version:** we target gum v0.14+ (current stable). The flags
   used (`choose`, `input`, `confirm`, `--no-limit`, `--header`,
   `--placeholder`, `--prompt`) are all stable since v0.10.

6. **huh v2 version:** the scaffold pins `charm.land/huh/v2 v2.0.0`
   (the stable line verified via context7 in CLAUDE.md). The form API
   used is `huh.NewForm(huh.NewGroup(field1, field2, ...)).Run()`.

7. **TTY + CI:** even if `$CI` is unset and stdin is a tty, an explicit
   `--no-interactive` flag wins. `--no-interactive` is the
   single-source-of-truth for "do not prompt".

8. **Cancellation in huh:** huh v2 returns an error from `form.Run()`
   on Esc / Ctrl-C. We map that to `*prompt.Canceled` and exit 130.
