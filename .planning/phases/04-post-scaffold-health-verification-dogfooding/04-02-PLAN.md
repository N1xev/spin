<objective>
Add `spin lint`, a thin cobra subcommand that wraps `golangci-lint` and follows the established `internal/wrap/*` pattern. Per CONTEXT D-04 + the Claude's Discretion section, `spin lint` mirrors `golangci-lint`'s own flag set via passthrough (no spin-specific flags). The companion `spin doctor --deep` already exec's `golangci-lint` inline (see Plan 04-01, `DeepLintCheck`); this plan provides the user-facing `spin lint` command as a separate, equally-thin wrapper. The two paths share the install-hint string.

Purpose: gives users a single command to run lint without remembering the binary name, and the hint makes it discoverable for users who don't have it installed.

Output: `internal/wrap/lint.go` + `cmd/lint.go` with unit + integration tests.
</objective>

Create `internal/wrap/lint.go`:
- Package `wrap`.
- Define an unexported package-level const `golangciLintInstallHint = "go install github.com/golangci/golangci-lint/v2/cmd/golangci-lint@latest"`. (Plan 04-01's `DeepLintCheck` should ideally reference this same const -- but Plan 04-01 is in a different file (`internal/doctor/checks.go`); to avoid cross-package coupling, define a const on each side. Both consts must contain the exact same string. Add a test in `internal/wrap/lint_test.go` asserting the const is non-empty and starts with `"go install "` and ends with `"@latest"`.)
- Define `Lint(args []string) error`:
  1. Call `exec.LookPath("golangci-lint")`. If found, call `runTool(path, args, nil)` and return its error.
  2. If not found, write a one-line message to `os.Stderr` of the form `hint: golangci-lint not found on $PATH; install with: <golangciLintInstallHint>` followed by a newline. Then return `fmt.Errorf("golangci-lint not found on $PATH")` so the cobra RunE propagates a non-zero exit through fang.
- `Lint` does NOT use `RunWithFallback` because there is no sensible fallback for a linter (a "fallback" that runs nothing or `go vet` would silently downgrade the user's lint signal). The explicit "not found → non-zero exit + hint" is the contract.
- The `args` parameter is forwarded verbatim -- `spin lint cache clean` becomes `golangci-lint cache clean`; `spin lint run --fix` becomes `golangci-lint run --fix`. There is no `spin lint`-specific argument parsing.

Create `internal/wrap/lint_test.go`:
- `TestLint_NotOnPath_ReturnsError` -- call `Lint(nil)` after renaming $PATH or pointing to an empty directory. Easiest approach: create a tempdir, set `os.Setenv("PATH", tmpdir)`, call `Lint(nil)`, restore $PATH in cleanup. Assert the returned error is non-nil and contains "golangci-lint not found". Assert os.Stderr captured a line starting with "hint: golangci-lint not found".
- `TestLint_StderrHint_HasInstallCommand` -- same as above, but assert the captured stderr line contains the literal string `go install github.com/golangci/golangci-lint/v2/cmd/golangci-lint@latest` (the install hint must mention the exact module path, not just "go install").
- `TestLint_ArgvPassThrough` -- call `Lint([]string{"version", "--help"})` with a fake `golangci-lint` binary that prints its argv. Construct the fake by writing a small shell script to a tempdir, prepend that tempdir to $PATH, and assert the fake's captured argv equals `["version", "--help"]` exactly. Restore $PATH in cleanup. This is the only test that exercises the "found on PATH" branch.
- `TestLintInstallHintConst_Shape` -- assert the const is non-empty, starts with `"go install "`, and ends with `"@latest"`. Cheap regression catcher if someone edits it to point to a different module.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User shell → `spin lint` | Untrusted `args []string` (user-controlled); forwarded verbatim to a child process. |
| `spin lint` → `golangci-lint` binary | The lint tool is first-party but receives untrusted argv. Trust: golangci-lint handles its own flag parsing safely. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-09 | Tampering | Argv forwarding to golangci-lint | mitigate | Use `exec.Cmd` with `cmd.Args = append([]string{name}, args...)` (already the runTool pattern) -- no shell interpolation. golangci-lint itself does its own flag validation. |
| T-04-10 | Information Disclosure | Stderr hint reveals install command | accept | The install command is the canonical public recipe (the project recommends it in CLAUDE.md). Disclosing it to a user running `spin lint` is the intended UX. |
| T-04-11 | Elevation of Privilege | A user could trick a script into running `spin lint` and expect no env changes | mitigate | `runTool` does NOT modify the inherited env. PATH, GOPATH, HOME, etc. all pass through. No secrets are read. |
| T-04-12 | Denial of Service | `golangci-lint` hangs on a large repo | accept | golangci-lint has its own timeout flags (`--timeout`); users can pass them. v1 of `spin lint` does not impose an additional timeout. |
| T-04-13 | Repudiation | Non-zero exit must reach the caller | mitigate | The `Lint` function returns the exec error directly; cobra RunE propagates it to fang. TestLintCmd_RunEForwardsArgs exercises the path. |
| T-04-SC | Tampering | No new package installs | n/a | No `go get` runs in this plan. golangci-lint is a developer tool, not a project dep. |
</threat_model>
