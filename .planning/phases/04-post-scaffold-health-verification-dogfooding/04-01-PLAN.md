<objective>
Implement `spin doctor`, a universal Go project health checker that runs four generic checks (Go version, tool presence, go.mod hygiene, CGO=0 buildability) plus an opt-in lint check under `--deep`. The CLI must support a styled human output, a stable JSON output, two-tier exit codes, `--strict` to upgrade warnings to failures, and `--fix` to apply safe repairs. No charm-specific checks (per CONTEXT D-01).

Purpose: HLTH-01; gives users a one-command way to know whether a generated or hand-rolled project is healthy before they invest time running build/test/lint.

Output: `internal/doctor/` package + `cmd/doctor.go` subcommand with full test coverage.
</objective>

Create `internal/doctor/doctor.go`:
- `Status` enum: `StatusPass`, `StatusWarn`, `StatusFail` (strings: "pass", "warn", "fail")
- `CheckResult` struct: `Name string`, `Status Status`, `Message string`, `Hint string` (optional install/fix hint)
- `Check` interface: `Name() string` and `Run(ctx context.Context) (CheckResult, error)`
- `Registry` type holding `[]Check`; `Register(c Check)` and `RunAll(ctx) []CheckResult` methods
- `RunOptions` struct: `Strict bool`, `Fix bool`, `Deep bool`, `Format string` ("human" or "json")
- Top-level `Run(ctx context.Context, opts RunOptions) (results []CheckResult, exitCode int)` orchestrator
- Exit-code logic: 0 if all pass or all warn (with Strict=false); 1 if any fail OR (Strict=true and any warn); the error from a Check.Run is treated as fail with err.Error() in Message

Create `internal/doctor/checks.go` with four checks:

1. `GoVersionCheck` -- runs `go version` via exec, parses the trailing version (strip "go" prefix), passes if `>= 1.23` (matches spin's own floor from CLAUDE.md), warns if `1.21-1.22`, fails if `< 1.21`. Use `golang.org/x/mod/semver` to compare (already a transitive dep).

2. `ToolPresenceCheck` -- for each of `air`, `prism`, `gofumpt`, `goimports`, run `exec.LookPath`. Pass if found with message "found at <path>"; warn if missing with Hint = install command (e.g. `go install github.com/air-verse/air@latest`). Missing tools are warn not fail (per D-05: --fix may install them).

3. `GoModHygieneCheck` -- uses `golang.org/x/mod/modfile` to parse `./go.mod`. Fails if no go.mod in cwd. Pass if the `go` directive is present and the `module` directive is present; warn if `// indirect` deps are listed but their module is also in the direct require block (a real but rare hygiene issue, easy to detect); pass otherwise. Look up the cwd by reading the current working directory with `os.Getwd()`; if no go.mod, fail with hint "run from a Go module root".

4. `CGOBuildCheck` -- runs `CGO_ENABLED=0 go build ./...` from cwd with a 60s timeout (use a small inline exec.CommandContext -- no need to import hooks.go's runCmdTimeout, just keep it local). Pass if exit 0; fail otherwise with combined stdout+stderr truncated to 1KB in Message.

5. `DeepLintCheck` (registered only when `opts.Deep` is true) -- runs `golangci-lint run ./...` via exec.LookPath. If golangci-lint is not on $PATH, warn with hint `go install github.com/golangci/golangci-lint/v2/cmd/golangci-lint@latest`. If found, pass if exit 0; fail otherwise. This is the implementation of D-04's "doctor --deep includes lint"; `cmd/lint.go` (Plan 04-02) is a separate, user-facing wrapper that mirrors this exec pattern.

For the `Fix` mode: the orchestrator's `Run` calls each Check's `Fix(ctx) error` if implemented and `opts.Fix` is true. Define a `Fixer` interface (optional, with `Fix(ctx) error`); only `GoModHygieneCheck` and `ToolPresenceCheck` implement it. `GoModHygieneCheck.Fix` runs `go mod tidy`. `ToolPresenceCheck.Fix` runs `go install <pkg>@latest` for each missing tool. `GoVersionCheck` and `CGOBuildCheck` are not fixable; passing `opts.Fix` to them is a no-op (log a "not fixable" message via the returned CheckResult.Hint).

- `TestDeepLintCheck_RegisteredWhenDeep` -- construct Registry with `opts.Deep=true`, assert DeepLintCheck is in the registered checks; with `opts.Deep=false`, assert it is not.

Create `internal/doctor/render.go`:

`RenderHuman(w io.Writer, results []CheckResult) error`:
- For each result, print a line: `  <icon> <name>  <message>` where icon is `✓` (pass), `!` (warn), `✗` (fail).
- Use lipgloss v2 styles per status: pass = green foreground, warn = yellow, fail = red. Apply styles to the icon glyph only (not the whole line) so the user's terminal width is not consumed by ANSI codes.
- After all checks, print a blank line and a summary line: `<N> passed, <M> warned, <K> failed`.
- If any result has a non-empty Hint, append a second indented line: `      hint: <hint>` (no styling -- keep it copy-pasteable).
- The output is written to `w` (typically `os.Stdout`); errors are returned for an io.Writer failure only.

`RenderJSON(w io.Writer, results []CheckResult) error`:
- Marshal `{"checks": [{"name": "...", "status": "pass|warn|fail", "message": "...", "hint": "..."}]}` to `w` via `encoding/json` with `json.NewEncoder(w).Encode(...)`.
- Use struct tags on a local `jsonCheck` view type (do not pollute the public `CheckResult` with json tags since the human renderer does not need them).
- The "checks" key and per-check keys are stable: do not rename them without a major version bump.
- Do NOT pretty-print (one line, no trailing newline issues). Use `json.Encoder` not `json.Marshal` so an io.Writer failure surfaces.

`FormatSelector(format string, results []CheckResult, w io.Writer) error`:
- If `format == ""` or `format == "human"`, call `RenderHuman`.
- If `format == "json"`, call `RenderJSON`.
- Else return `fmt.Errorf("doctor: unknown --format %q (want human or json)", format)`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User shell → `spin doctor` | Untrusted cwd (the user may run from anywhere); the CLI must not assume a go.mod is present. |
| `spin doctor` → local Go toolchain | The CLI shells out to `go`, `gofumpt`, `golangci-lint`. Trust: these tools are first-party or installed by the user. |
| `spin doctor` → file system | Reads `go.mod` and the project tree; never writes except when `--fix` is set. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | `GoModHygieneCheck` parses go.mod | mitigate | Use `golang.org/x/mod/modfile.Parse` (the official parser) -- does not eval, does not shell out. |
| T-04-02 | Tampering | `CGOBuildCheck` runs `go build` | mitigate | Static argv (no interpolation), `CGO_ENABLED=0` env passed via cmd.Env (not shell), 60s context timeout, output truncated to 1KB. |
| T-04-03 | Tampering | `ToolPresenceCheck` shells out to detect tools | mitigate | `exec.LookPath` does not execute; only checks $PATH resolution. No exec at this stage. |
| T-04-04 | Information Disclosure | JSON output may leak paths or env | accept | JSON output contains only check name, status, message, hint -- no env vars, no secret material. Paths limited to the cwd basename (the go.mod's module path). |
| T-04-05 | Elevation of Privilege | `--fix` runs `go mod tidy` and `go install` | mitigate | Tidy is read-mostly (may add go.sum entries but does not change code). `go install` only fires for the 4 whitelisted tools (air, prism, gofumpt, goimports) with pinned install commands in code -- no user input is interpolated. |
| T-04-06 | Denial of Service | CGO build on huge monorepo | mitigate | 60s timeout via `exec.CommandContext`. DeepLintCheck uses the same pattern. |
| T-04-07 | Repudiation | Exit code does not match observed behavior | mitigate | Tests assert exit codes for all-pass, any-fail, and warn-under-strict. CI binary exit is the contract. |
| T-04-08 | Tampering (slopcheck) | No new package installs in this plan | n/a | Plan adds no new direct dependencies; golang.org/x/mod and charm.land/lipgloss/v2 are already in go.mod. |
</threat_model>
