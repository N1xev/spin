Create `scripts/dogfood.sh`:
- Shebang: `#!/usr/bin/env bash`.
- `set -euo pipefail` at the top — same style as the existing scripts/check-*.sh files.
- Resolve the repo root: `REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"`.
- `BIN="$REPO_ROOT/bin/spin"`; `WORK="$(mktemp -d)"`; trap `rm -rf "$WORK"` EXIT (always clean up).
- `echo "==> Building spin"`. `cd "$REPO_ROOT" && go build -o "$BIN" .` (fail loud if the build itself is broken — the whole point of the dogfood job is to catch scaffolder regressions).
- `echo "==> Scaffolding fixture project"`. `cd "$WORK" && "$BIN" new spin --cli --cobra --fang --module github.com/example/spin-fixture --quiet`. The `--module github.com/example/spin-fixture` override avoids colliding with the real `github.com/example/spin` module path (the scaffold runs in a tempdir, but the embedded `go.mod` is what matters for `go mod tidy`).
- `echo "==> Running go mod tidy"`. `cd "$WORK/spin" && go mod tidy` (with `CGO_ENABLED=0` env).
- `echo "==> Running go build"`. `cd "$WORK/spin" && CGO_ENABLED=0 go build ./...` — the static-build smoke test.
- `echo "==> Running go test"`. `cd "$WORK/spin" && go test ./... -count=1` — confirms the generated test suite passes.
- `echo "==> Running v1-leak grep"`. `bash "$REPO_ROOT/scripts/check-v1-leaks.sh" "$WORK/spin"` — runs the same CI grep suite that runs in the regular `test` job. Catches a regression where a template change accidentally introduces a v1 charmbracelet import.
- `echo "==> dogfood passed"`. Exit 0.

If any step fails, the script must print the failing step name and the last 50 lines of its output. Implement this by capturing each step's output to a tmpfile and dumping it on failure:
```bash
run_step() {
  local name="$1"; shift
  echo "==> $name"
  if ! "$@" > "$WORK/${name// /_}.log" 2>&1; then
    echo "FAIL: $name" >&2
    tail -50 "$WORK/${name// /_}.log" >&2
    exit 1
  fi
}
```
Wrap each step in `run_step "Building spin" bash -c "cd $REPO_ROOT && go build -o $BIN ."` (or similar).

Make the script executable: `chmod +x scripts/dogfood.sh` (the executor should do this; the file mode is set in the Write tool call). Verify with `ls -l scripts/dogfood.sh` after write — first char should be `-rwxr-xr-x`.

Add a small comment block at the top explaining: "This is the local-runnable form of the GitHub Actions `dogfood` job. If you change the scaffolder templates or any cmd/* file, run this before pushing; the CI job will run the same pipeline."

Also: the script should not assume `golangci-lint` is installed (the v1-leak grep does not need it).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → scripts/dogfood.sh | Untrusted in the sense that the script runs with the runner's filesystem and network. The script does not exfiltrate data; it only builds and tests in a tempdir. |
| Dogfood job → spin scaffolder | The script invokes the spin binary built from the same commit, so the scaffolder being tested is exactly the code under review. No supply-chain risk (the binary is built from `actions/checkout` of the same commit, not `go install`). |
| Scaffold result → `go build` + `go test` | The generated fixture project is run with full network access (for `go mod tidy` to fetch deps). This is the same surface as the existing `scaffold_e2e_test.go` integration test. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-30 | Tampering | Malicious PR modifies scripts/dogfood.sh to exfiltrate secrets | mitigate | The workflow uses `permissions: { contents: read }` (no `actions: write`, no `packages: write`). Even if the script is compromised, it cannot push code or post status. The runner has no GitHub token to use beyond the default GITHUB_TOKEN, which only has read access on the checkout. |
| T-04-31 | Tampering | The `dogfood` script could be modified to skip the smoke test (e.g. `exit 0` at the top) | mitigate | The workflow's `paths:` filter includes `scripts/dogfood.sh` itself, so changes to the script trigger a re-run. A reviewer sees the diff in the PR. The script's `set -euo pipefail` prevents a silent `true` from masking failures (though a determined attacker could still `set +e` first — but that diff is also visible). |
| T-04-32 | Tampering | Generated fixture pulls arbitrary modules from proxy.golang.org | accept | This is the same surface as the existing local `scaffold_e2e_test.go`. The Go toolchain's sum.golang.org verification protects against malicious module content. The runner has no access to real secrets (just `GITHUB_TOKEN` with read-only permissions). |
| T-04-33 | Denial of Service | The dogfood job runs every PR that touches templates; could be slow | mitigate | The job uses `ubuntu-latest` (fast runners, ~30s typical for a small scaffold). The script has no unbounded loops or waits. If a perf issue arises, add a `concurrency:` group to the workflow to cancel in-progress runs on the same PR. |
| T-04-34 | Repudiation | A failing dogfood job must produce enough output to debug | mitigate | The `if: failure()` upload-artifact step makes the binary and logs available as a downloadable artifact. The script's `run_step` helper also prints the last 50 lines of any failing step. |
| T-04-SC | Tampering (slopcheck) | No new direct dependencies | n/a | Only GitHub Actions first-party actions are used: `actions/checkout@v4`, `actions/setup-go@v5`, `actions/upload-artifact@v4`. All are official, widely-used, and verified by the GitHub ecosystem. No `go install` of unknown packages runs in CI. |
</threat_model>
