# Phase 4: Post-Scaffold Health + Verification + Dogfooding - Context

**Gathered:** 2026-06-04
**Status:** Ready for planning

## Phase Boundary

Two new universal Go subcommands (`spin doctor`, `spin update`) plus a `spin lint` wrapper for `golangci-lint`. A CI dogfood job in spin's own build matrix. Strip `// generated by spin X.Y.Z` headers from all generated files. Drop `spin add <lib>` from this phase.

Both `spin doctor` and `spin update` are universal Go tooling — they work on any project with a `go.mod`, not just spin-scaffolded ones. The dogfood job proves the scaffolder on its own codebase.

## Implementation Decisions

### `spin doctor` (HLTH-01)

- **D-01:** Four universal checks, all general-purpose (no charm-specific validation): Go version compatibility, tool presence (air, prism, gofumpt, goimports), go.mod hygiene (Go directive present, no stray entries), `CGO_ENABLED=0 go build ./...` buildability.
- **D-02:** Output `--format human|json`. Human (default) is lipgloss-styled check-by-check with pass/fail and install hints. JSON for CI/scripts.
- **D-03:** Two-tier exit: 0 = all pass, 1 = any failure. Warnings (e.g. gofumpt missing with `--no-strict` previously used) keep exit 0; `--strict` upgrades warnings to exit 1. Matches `go vet` / `golangci-lint` convention.
- **D-04:** `spin doctor --deep` includes lint. New `spin lint` command wraps `golangci-lint`. Keeps the base doctor fast (sub-5s).
- **D-05:** Read-only default. `spin doctor --fix` runs safe repairs: `go mod tidy`, re-add `go` directive, attempt `go install` for missing tools.

### `spin update` (HLTH-03, rewritten)

- **D-06:** Universal Go dep updater. No spin-project detection. Works on any `go.mod`.
- **D-07:** Direct deps from the `require` block (excluding `// indirect`) by default. `--all` flag includes indirect deps.
- **D-08:** Three columns per dep: `old` = currently pinned version, `newStable` = highest semver-descending with no pre-release suffix (`-beta`, `-rc`, `-alpha`, `-pre`), `newLatest` = highest semver-descending including pre-releases.
- **D-09:** Single `huh v2` form with one `Select` field per dep. Options per field: `[Skip, newStable, newLatest]`. Default per field: `newStable`. One submit applies the entire batch atomically.
- **D-10:** Apply mechanics: `go get module@version` for each selected dep → `go mod tidy` once → `go build ./...` smoke test. Exit non-zero on build failure. Does NOT run `go test ./...` (user's test suite is the user's concern).

### Dropped from Phase 4

- **D-11:** HLTH-02 (`spin add <lib>`) — explicitly removed. Library injection is its own phase if needed later; not in Phase 4.
- **D-12:** HLTH-04 (file headers for traceability) — explicitly removed. `// generated by spin X.Y.Z` and `<!-- AUTOGENERATED by spin X.Y.Z -->` markers are stripped from all generated files (Go, YAML, TOML, AGENTS.md, etc). The "owned-by-spin" signal concept from Phase 3-04 is retired. `spin update` no longer needs to identify spin-owned files because it's universal.

### Dogfood (ROADMAP success criterion #5)

- **D-13:** Add a `dogfood` CI job in the spin repo's build matrix. Runs `spin new spin --cli --cobra --fang` in a fresh temp dir, then `go build ./...` + `go test ./...` on the result. Smoke test for the scaffolder. Triggers on changes to `internal/scaffold/templates/**` and `cmd/`.

### Claude's Discretion

- Long-form huh behavior for `spin update` with 20+ deps: rely on huh's built-in scroll; defer pagination/grouping until a real user hits a long form.
- `go list -m -versions` vs direct `proxy.golang.org` HTTP for resolving `newStable`/`newLatest`: research phase should pick the cleaner one; both work.
- The `spin lint` command's flag set mirrors `golangci-lint`'s (`run`, `cache clean`, etc.) by passing through; no spin-specific flags needed.

## Existing Code Insights

### Reusable Assets
- `internal/wrap/detect.go` — `Detect(bin string) (string, bool)` returns path + availability. Use this exact API for doctor's tool-presence check.
- `internal/scaffold/resolve.go` — go.mod parsing. Extract pattern for `spin update` to read `require` block.
- `internal/scaffold/project.go` — `Project` struct has `Libs`, `Module`, `Name`, `Type` fields. Not directly used by update (universal tool) but the resolution helpers may be.

### Established Patterns
- **Wrapper pattern**: every wrapper is `internal/wrap/<name>.go` (impl) + `<name>_test.go` (test). The detect.go pattern unifies tool presence across wrappers.
- **Command pattern**: every CLI subcommand is `cmd/<name>.go` with `init() { rootCmd.AddCommand(<name>Cmd) }`. New subcommands follow this.
- **Post-scaffold hook pattern**: `internal/scaffold/hooks.go` runs validation/build after scaffold. The dogfood CI job inverts this (run the scaffolder, then build the result).
- **Huh v2 in-process prompts** (Phase 3): `internal/prompt/huh.go` (or similar) — wire `spin update`'s confirm form to the same huh setup.

### Integration Points
- `cmd/root.go` — add three new subcommands: `doctor`, `update`, `lint`.
- `internal/scaffold/templates/**` — strip `// generated by spin {{.SpinVer}}` lines and equivalent markers.
- `internal/scaffold/template.go` (or where `SpinVer` is defined) — drop the variable if no template uses it post-strip.
- New CI: `.github/workflows/dogfood.yml` (or add `dogfood` job to existing workflow).

## Specific Ideas

- `spin update` and `spin doctor` should be reachable from any directory that has a `go.mod` (universal). They `os.Chdir` (or operate on the current dir) to find the nearest `go.mod` going up — same pattern as `go list` / `go test ./...`.
- The huh form for `spin update` should render rows like: `charm.land/bubbletea/v2  v2.0.0  →  v2.0.7  →  v2.0.8-beta.2  [newStable ▾]`. Lipgloss for the columns.
- `spin doctor` JSON output should be a stable schema (e.g. `{ "checks": [{ "name": "go-version", "status": "pass"|"warn"|"fail", "message": "..." }] }`) so CI scripts can parse it.
- `spin doctor --deep` could parallelize lint+build via errgroup; for v1 just sequential is fine.
